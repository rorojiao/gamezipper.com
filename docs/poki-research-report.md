# Poki.com 设计体系研究报告

**报告用途**：GameZipper.com 重新设计参考文档
**研究对象**：Poki.com 全球最大在线游戏平台
**报告语言**：简体中文

---

## 1. Poki 网站概述与定位

### 1.1 平台定位

Poki.com 是全球访问量最大的免费在线游戏平台之一，专注于为用户提供无需下载、无需注册的即时可玩网页游戏。其核心定位可以归纳为以下几个维度：

- **用户价值**：一键点击即可开始游戏，消除一切游戏障碍
- **内容策略**：聚合全球优秀 HTML5 游戏开发者作品，通过高质量筛选机制保证内容品质
- **商业模型**：以游戏加载间隙的原生广告为主要营收来源，用户体验优先于广告展示
- **品牌调性**：年轻、友好、干净、现代——强调"游戏本身即界面"

### 1.2 平台规模与影响力

根据研究数据，Poki.com 首页承载了约 162 个游戏瓦片，详情页推荐 104 款相关游戏，平台收录游戏覆盖 20 个核心分类。这种大规模内容聚合配合极简发现界面，形成了其独特的"发现式浏览"体验。Poki 页面结构极其精简，HTML 总量仅 385KB，却承载了 73 个脚本标签和 212 个图片资源，体现了极高的工程化水平。

### 1.3 设计理念核心

Poki 设计哲学最显著的特征是 **"零文本发现"（Zero-Text Discovery）**——游戏卡片不显示游戏名称，游戏的视觉缩略图本身就是交互界面。这一理念颠覆了传统游戏门户"标题+封面"的展示范式，将发现游戏的成本降至最低。用户通过视觉直觉而非文字搜索来探索内容，这与 Instagram/Pinterest 的图片流体验异曲同工。

## 2. 设计系统详解

### 2.1 色彩体系

#### 2.1.1 背景色

Poki 的背景系统采用渐进式色彩策略，不同页面区域使用不同明度的色彩引导用户视线：

| 区域 | 色值 | 用途 |
|------|------|------|
| 主背景渐变起点 | `#E6F9FF` | 页面顶部渐变色，低饱和度天蓝 |
| 主背景渐变终点 | `#FFFFFF` | 页面主体背景，白色 |
| 首屏区块背景 | `rgb(131, 255, 231)` | Teal/Mint 薄荷绿，用于首屏强调区块 |

这种背景处理方式在视觉上营造出"从清新空气过渡到纯净画布"的感受，薄荷绿首屏区块作为视觉锚点立即传达平台的年轻化品牌调性。

#### 2.1.2 品牌色与强调色

| 用途 | 色值 | 说明 |
|------|------|------ |
| 品牌主色 | `#35C6B5` | Teal/Mint 薄荷绿，用于 Logo、按钮、强调元素 |
| 正文文字 | `#002B50` | 深海军蓝，rgb(0, 43, 80)，保证可读性的同时与品牌色形成互补 |
| 卡片阴影 | `rgba(93, 107, 132, 0.3)` | 蓝灰色半透明，用于游戏卡片投影 |

#### 2.1.3 高饱和度卡片色

Poki 的游戏卡片在特定分类下会使用高饱和度背景色来增强视觉区分度：

| 色值 | 视觉感受 | 适用场景 |
|------|----------|----------|
| `#FFD500` | 明黄 | 强调类、热门游戏 |
| `#9B59B6` | 紫罗兰 | 创意类、休闲游戏 |
| `#E74C3C` | 珊瑚红 | 动作类、竞技游戏 |

这些高饱和色彩与整体低饱和背景形成强烈对比，在灰度低的卡片海洋中制造视觉焦点，引导用户注意力。

### 2.2 字体系统

#### 2.2.1 字体族

```
"Proxima Nova", "Open Sans", "Gill Sans MT", Arial, sans-serif
```

Poki 选择了 Proxima Nova 作为首选品牌字体，这是一款几何感与现代感兼备的无衬线字体，被广泛用于 Spotify、GitHub 等科技公司产品中。字体回退链设计合理：Proxima Nova 不存在时回退到 Open Sans（Google Fonts，广泛支持），再回退到系统 Arial，确保跨平台一致性。

#### 2.2.2 字号体系

- **基础字号**：`16px`（用户代理样式默认值）
- **标题层级**：通过字重（font-weight）和字号的组合来建立视觉层级，未使用过度夸张的标题尺寸
- **风格特点**：整体字号偏保守，强调可读性和舒适度，避免过度设计

#### 2.2.3 字体风格

整体字体风格为 **Clean、Rounded、Friendly**——干净但不生硬，圆润但不幼稚。这种风格选择与其"零门槛游戏平台"的定位高度契合，字体不传递任何压迫感或专业门槛。

### 2.3 间距系统

Poki 的间距系统以 **8px 为基础单位**进行设计，这从游戏网格系统的 16px gap（8px 的 2 倍）可以验证。关键间距参数：

- **网格 gap**：`16px`（游戏卡片之间的水平和垂直间距）
- **卡片尺寸**：`94px × 94px`（正方形，略偏方形而非正方形，视觉上更舒展）
- **内边距**：首屏区块采用较大留白，内容区域紧凑排列

### 2.4 圆角系统

| 元素 | 圆角值 | 说明 |
|------|--------|------|
| 游戏卡片 | `16px` | 较大的圆角值，传达友好、年轻感 |
| 缩略图 | `16px` | 与卡片保持一致，圆角与方形之间取得平衡 |
| 分类卡片 | 白色矩形 | 分类卡片为直角矩形，与游戏卡片形成材质对比 |

16px 的圆角是一个中间值——既不像 4px 那样生硬，也没有 24px 那样过于圆润，保持了专业感与亲和力的平衡。

### 2.5 阴影系统

#### 2.5.1 卡片阴影

```
box-shadow: rgba(93, 107, 132, 0.3) 0px 7px 10px 4px
```

这是一个多层阴影组合：
- **扩散半径**：`7px`（垂直方向）
- **模糊距离**：`10px`
- **偏移**：`4px`（水平方向）
- **色彩**：`rgba(93, 107, 132, 0.3)` — 蓝灰色，30% 透明度

这种阴影效果在白色背景上产生"卡片浮起"的视觉感受，深度感适中，既不刺眼也不廉价。

#### 2.5.2 阴影设计原则

Poki 的阴影策略是"轻量感"——阴影用于建立层级关系，而非制造戏剧性效果。蓝灰色调（而非纯黑）使阴影融入整体冷色调配色体系，避免与薄荷绿品牌色冲突。

### 2.6 动效过渡

```
transition: transform 0.6s cubic-bezier(0.25, 0.1, 0.25, 1)
```

卡片 hover 过渡使用较慢的 0.6s 过渡时间，配合平滑的贝塞尔曲线 cubic-bezier(0.25, 0.1, 0.25, 1)。这个曲线特点是开始快、结束慢，给用户一种"缓入缓出"的优雅感受。0.6s 的时长偏长，但正好与其"发现式浏览"的用户行为节奏匹配——用户 hover 时会稍作品味，而非快速划过。

## 3. 首页布局分析

### 3.1 整体架构

Poki 首页采用 **全宽发现（Full-Width Discovery）** 模式，没有任何传统侧边栏，页面从头到尾是一段连续的内容流。整体布局结构如下：

```
┌─────────────────────────────────────────────────────┐
│  [浮动 Logo + 搜索图标 + 头像图标]   ← 浮动导航 (左上角) │
├─────────────────────────────────────────────────────┤
│                                                     │
│     首屏区块 (Teal/Mint 背景)                       │
│     — 品牌表达区，通常是特色游戏或品牌标语            │
│                                                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│     主游戏网格 (6列 × N行)                          │
│     — 94×94px 卡片，16px gap                       │
│     — 162 个游戏瓦片                               │
│     — 无标题、无文字，游戏封面即全部界面             │
│                                                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│     分类导航区                                       │
│     — 20个分类，白色矩形卡片带图标                   │
│     — 分类: two-player, shooting, car, action...     │
│                                                     │
├─────────────────────────────────────────────────────┤
│  ════════════ Teal 波浪过渡 ══════════════════════ │
├─────────────────────────────────────────────────────┤
│     页脚 (4列布局)                                   │
│     Brand | Popular | Support | Corporate           │
│     Social: TikTok, Instagram, YouTube              │
└─────────────────────────────────────────────────────┘
```

### 3.2 导航系统

#### 3.2.1 浮动垂直导航

Poki 的导航设计极具特色——**不是横向顶部导航栏，而是左上角浮动的垂直 Logo 组**：

- **Logo 形式**：`白色 "poki" 字样 + 圆形薄荷绿背景`
- **搜索**：仅一个图标，点击后展开全屏搜索覆盖层
- **用户入口**：仅一个头像图标
- **无分类链接**：导航上没有任何文字链接，分类完全集成到页面底部的分类卡片区

这种设计彻底解放了页面顶部空间，用户视线可以完全沉浸在游戏内容中。

#### 3.2.2 导航设计哲学

传统游戏门户（如 4399）的导航通常是顶部横条 + 多级分类菜单，这种设计在内容密度上有效但视觉上具有压迫感。Poki 的浮动 Logo 设计将品牌标识"轻量化"处理——用户记得"去 Poki 玩游戏"，而不是"去 Poki 的 XXX 分类找游戏"。

### 3.3 游戏网格布局

#### 3.3.1 网格参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 列数 | 6 列 | 桌面端 6 列游戏网格 |
| 卡片尺寸 | 94×94px | 正方形游戏缩略图 |
| 间距 | 16px | 水平和垂直 gap 相同 |
| 圆角 | 16px | 缩略图和容器圆角统一 |
| 阴影 | `rgba(93,107,132,0.3) 0px 7px 10px 4px` | 卡片浮起效果 |
| 卡片数量 | 约 162 个 | 首页承载量 |

#### 3.3.2 CSS Grid 实现

```css
display: grid;
grid-template-columns: repeat(6, 94px);
gap: 16px;
```

6 列 94px 的设计使得网格总宽度约为 `6 × 94 + 5 × 16 = 564 + 80 = 644px`，这个宽度在主流显示器（1280px+）上左右留有充足留白，视觉上居中但不局促。

#### 3.3.3 零文本发现机制

游戏卡片**不显示游戏名称**，这是 Poki 区别于所有同类平台的核心设计决策。这一设计的基础假设是：

1. 游戏封面图（Logo）本身就是最有效的游戏标识
2. 用户在浏览游戏时，视觉驱动优先于文字搜索
3. 减少文字信息可以极大提升页面内容密度和视觉简洁度

当用户 Hover 卡片时，可能会有游戏名称的 tooltip 出现，但默认状态下页面是纯图片流。

### 3.4 分类系统

#### 3.4.1 分类卡片设计

20 个游戏分类以**白色矩形图标卡片**的形式排列在游戏网格下方，每个分类包含一个简洁的图标和分类名称。20 个分类为：

`双人` · `射击` · `汽车` · `动作` · `Roblox` · `Minecraft` · `多人` · `技能` · `IO` · `冒险` · `益智` · `脑力` · `换装` · `体育` · `数学` · `女孩` · `格斗` · `模拟` · `Obby` · `平台`

这些分类覆盖了主流 HTML5 游戏的所有核心类型。值得注意的是 Poki 直接纳入了 `Roblox` 和 `Minecraft` 这类寄生型分类（这两个实际上是游戏平台/生态，而非游戏类型），说明其分类体系是**用户导向**而非**严格学术分类**。

#### 3.4.2 分类入口的作用

分类卡片区是首页的"导航终点"——当用户浏览完游戏网格后，向下滚动会自然到达分类区，选择感兴趣的分类进入筛选视图。这种"游戏探索 → 分类筛选"的双层发现路径符合"先浏览后决策"的用户心理。

### 3.5 页脚系统

#### 3.5.1 波浪过渡

Poki 在内容区和页脚之间使用了 **Teal 色彩波浪 SVG 过渡**，视觉上柔和地区分了内容层和底栏，同时强化品牌色彩。

#### 3.5.2 页脚四列布局

| 列 | 内容 |
|----|------|
| Brand | 关于我们、品牌故事 |
| Popular | 热门游戏入口 |
| Support | 帮助中心、联系我们 |
| Corporate | 商务合作、广告合作 |

#### 3.5.3 社交媒体

平台在页脚放置了 TikTok、Instagram、YouTube 三个社交渠道的入口，这三个平台的选择（缺少 Facebook/Twitter）反映了 Poki 对年轻用户群体聚集地的判断。

## 4. 游戏详情页布局分析

### 4.1 概述

以 Subway Surfers（地铁跑酷）作为分析样本，Poki 的游戏详情页是典型的 **"游戏优先"** 布局。

### 4.2 视觉结构

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│   ┌────────────────────────────────────────────────┐     │
│   │                                                │     │
│   │           游戏 iframe 嵌入区                    │     │
│   │        (836×470px, 1280×577 视口)              │     │
│   │                                                │     │
│   │   [广告覆盖层: "We'll be back after this break"]│     │
│   │                                                │     │
│   └────────────────────────────────────────────────┘     │
│                                                          │
│   游戏标题    开发者 "by SYBO"                           │
│   17.7M 👍  3.1M 👎  [举报bug]  [全屏]                   │
│                                                          │
│   ════════════ 推荐游戏网格 (104款) ══════════════════   │
│   [游戏卡片] [游戏卡片] [游戏卡片] ...                    │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 4.3 游戏框架

- **游戏视口尺寸**：`836×470px`（在 1280×577 的页面视口中）
- **宽高比**：约 16:9，符合主流游戏和视频内容比例
- **全屏按钮**：提供全屏游戏体验入口
- **报告 Bug 按钮**：用户反馈机制，体现平台对游戏质量的关注

### 4.4 游戏信息区

游戏详情页的游戏信息区极为精简：

- **标题**：游戏名称（如 "Subway Surfers"）
- **开发者**：前缀 "by " + 开发者名称（如 "by SYBO"）
- **互动数据**：点赞数 17.7M、点踩数 3.1M（以图标+数字形式呈现）
- **操作按钮**：举报 Bug、全屏

点赞/点踩功能的存在说明 Poki 建立了一套**用户评价反馈系统**，这对于游戏质量筛选和推荐算法都有重要价值。

### 4.5 推荐系统

详情页底部展示了 **104 款推荐游戏**，这个数量远大于首页的 162 款推荐量，说明详情页的推荐逻辑更偏向"同类延伸"而非"精选发现"。推荐游戏以相同的 94×94px 卡片网格呈现，保持了与首页的视觉一致性。

### 4.6 广告策略

游戏加载前/间隙会显示全屏广告，广告文案为 **"We'll be back after this short break"**（我们将在这个短暂间隙后回来）。这种文案处理方式将广告体验拟人化，将广告间隙定义为"平台暂停"而非"广告轰炸"，有效降低了用户对广告的负面情绪。

## 5. 交互体验分析

### 5.1 搜索体验

#### 5.1.1 图标式搜索

搜索入口在导航中**仅以图标形式存在**（放大镜图标），不显示搜索框或文字提示。这种设计的好处是：

- 视觉干扰极低，用户不会被搜索框分散注意力
- 保持整体界面简洁性
- 符合"零文本发现"的整体设计哲学

#### 5.1.2 全屏搜索覆盖层

点击搜索图标后，搜索覆盖层以**全屏形式**展开，这种模式的优势是：

- 搜索区域不受页面布局限制，可以最大化利用屏幕空间
- 全屏暗色背景将用户注意力完全聚焦到搜索框
- 适合触屏设备和键盘输入

### 5.2 Hover 交互

#### 5.2.1 卡片 Hover 效果

游戏卡片的 Hover 状态使用以下过渡效果：

```css
transform: scale(1.05);  /* 或类似轻微放大效果 */
transition: transform 0.6s cubic-bezier(0.25, 0.1, 0.25, 1);
```

关键设计考量：

- **放大比例**：轻微放大（约 1.05x）而非剧烈变换，视觉反馈微妙但可感知
- **过渡时长**：0.6s 的慢速过渡适合浏览节奏，用户有足够时间欣赏目标游戏
- **贝塞尔曲线**：cubic-bezier(0.25, 0.1, 0.25, 1) 的特点是缓入缓出，动画不突兀

#### 5.2.2 Hover 效果的设计意图

由于游戏卡片没有文字标题，Hover 效果是用户确认目标游戏的**唯一反馈机制**。这要求 Hover 效果必须：

1. 足够明显，让用户知道"我在选这个"
2. 不能过于夸张，避免干扰周围卡片
3. 过渡平滑，不打断浏览节奏

### 5.3 加载体验

#### 5.3.1 游戏加载流程

```
用户点击卡片 → 游戏 iframe 加载 → 广告间隙（如有）→ 游戏可玩
```

Poki 的游戏加载采用 iframe 嵌入模式，用户无需跳转到新页面，即可在当前页面无缝开始游戏。

#### 5.3.2 加载感知优化

- 详情页游戏框架尺寸为 836×470px，较大的游戏窗口让用户在等待加载时有一个明确的视觉焦点
- 广告覆盖层用友善文案降低等待焦虑

### 5.4 广告策略分析

#### 5.4.1 广告 Light 原则

Poki 最重要的商业设计原则是 **Ad-Light Browsing（广告轻量浏览）**：

- **首页零广告**：用户在浏览游戏库时完全看不到任何广告banner
- **广告仅在游戏加载时出现**：广告价值与用户意图高度匹配（用户正准备玩游戏）
- **广告形式**：游戏加载前的插屏式广告，而非 banner 或弹出式

这种策略的商业逻辑是：用户在"准备玩游戏"的心理状态下对广告的容忍度最高，广告效果也最好。

#### 5.4.2 广告文案设计

"We'll be back after this short break" 这句文案的设计意图：

- 用"我们"（平台方）而非"广告"，将商业内容拟人化
- "short break"而非"ad"避免直接触发用户的广告排斥心理
- 承诺回归（"back"）暗示游戏内容即将到来，给用户期待感

### 5.5 整体交互节奏

Poki 的交互设计遵循 **"Scroll-to-Discover"（滚动发现）** 模式：

1. 用户到达首页 → 视觉立即被首屏特色内容捕获
2. 向下滚动 → 大量游戏卡片涌入视野，无尽内容流
3. Hover 卡片 → 微妙放大反馈，确认选择
4. 点击卡片 → 直接进入游戏详情页或开始游戏
5. 游戏结束后 → 返回首页继续浏览

整个体验是**低摩擦、无断点**的，用户始终在"发现-选择-体验"的正向循环中。

## 7. SEO 策略

### 7.1 技术 SEO

Poki 作为大型内容平台，技术 SEO 方面应具备以下特征：

- **语义化 HTML**：使用正确的标签结构（H1-H6、article、section、nav 等）
- **快速加载**：HTML 仅 385KB，说明在资源压缩和加载优化上做了大量工作
- **图片优化**：使用 CDN 分发，图片支持多种尺寸参数（width=94, width=204 等）
- **Meta 标签**：每个游戏详情页应有独立的 title、description、OG 标签

### 7.2 内容 SEO

游戏平台的内容 SEO 核心在于：

- **游戏名称**：URL、title、description 中包含游戏名称
- **分类页面**：每个分类有独立的可索引页面（如 `/category/shooting`）
- **开发者页面**：按开发者聚合游戏（详情页显示 "by SYBO"）
- **长尾关键词**：游戏攻略、评测类内容可以覆盖长尾搜索需求

### 7.3 图片 SEO

由于 Poki 的游戏卡片使用"零文本发现"模式，图片的 alt 标签和文件命名对 SEO 至关重要：

- 图片 URL 格式：`{hash}/{game-name-logo}.png` — 游戏名称出现在 URL 中
- CDN 参数：`q=78,scq=50` 等质量参数用于控制图片压缩比

### 7.4 结构化数据

建议为游戏详情页添加以下 Schema.org 结构化数据：

- `VideoGame` 类型：游戏名称、开发者、发布日期、评分
- `AggregateRating`：点赞/点踩数据可映射为评分
- `BreadcrumbList`：首页 > 分类 > 游戏

## 9. GameZipper 重新设计建议

### 9.1 设计系统建立

#### 9.1.1 色彩体系建议

**建议采用薄荷绿作为主品牌色**（参考 Poki #35C6B5），搭配深蓝文字色（#002B50），背景使用低饱和蓝白渐变。这一配色体系在游戏平台领域能够有效传达"年轻、干净、友好"的品牌调性。

**执行步骤**：
1. 定义品牌主色、辅助色、背景色、文字色的色值规范
2. 建立卡片高饱和色板（用于分类区分或活动强调）
3. 建立语义化色彩 token（`color-primary`, `color-bg`, `color-text` 等）

#### 9.1.2 字体系统建议

**建议引入 Proxima Nova 或类似几何无衬线字体**（如 Montserrat、Poppins）作为品牌字体，建立 3-4 级字号体系。桌面端基础字号建议保持 16px，移动端可微调为 14-15px。

**执行步骤**：
1. 通过 Google Fonts 引入开源等效字体（如 Poppins）
2. 定义标题（H1-H4）、正文、辅助文字的字号和字重规范
3. 建立行高和字间距规范（通常行高 1.5 左右）

#### 9.1.3 间距与圆角规范

**建议以 8px 为基础间距单位**，建立 8/16/24/32/48 的间距 scale。卡片圆角建议统一使用 12-16px，与 Poki 保持一致。

**执行步骤**：
1. 在 CSS 变量或 design token 系统中定义间距 scale
2. 建立组件级圆角规范（按钮、卡片、输入框各自独立）
3. 建立阴影规范（建议使用与 Poki 相似的蓝灰半透明阴影）

#### 9.1.4 动效规范

**建议建立统一的过渡动画规范**：
- 默认过渡：`0.3s ease`（常规交互）
- 卡片 Hover：`0.6s cubic-bezier(0.25, 0.1, 0.25, 1)`
- 页面切换：`0.3s ease-out`
- 搜索展开：`0.25s ease-in-out`

### 9.2 导航系统重构

#### 9.2.1 浮动导航实现

**建议将顶部横向导航栏改为左上角浮动 Logo 组**：
- Logo 放置在左上角，使用圆形薄荷绿背景 + 白色文字
- 搜索仅保留图标，点击展开全屏搜索覆盖层
- 移除导航栏中的分类文字链接（将分类入口迁移到页面底部卡片区）
- 保留用户头像入口（登录/未登录状态）

**执行步骤**：
1. 设计浮动导航组件（固定定位 left: 24px, top: 24px）
2. 实现搜索图标的点击行为（全屏遮罩 + 居中搜索框）
3. 移除顶部导航栏的分类链接
4. 设计移动端导航的响应式策略（可能需要汉堡菜单）

#### 9.2.2 分类区重新设计

**建议在页面底部增加图标型分类导航区**：
- 20 个分类以 2 行 × 10 列（或响应式调整）的白色矩形卡片呈现
- 每个卡片包含分类图标 + 分类名称
- 分类图标建议使用 Lucide Icons 或 Font Awesome 等开源图标库

### 9.3 游戏网格重构

#### 9.3.1 零文本发现模式

**建议逐步过渡到"零文本发现"模式**，具体执行策略：

**方案 A（激进）**：完全移除游戏卡片上的标题文字，仅展示游戏缩略图
**方案 B（渐进）**：缩小标题文字（12px, 灰色），缩略图占据卡片主导地位

**执行步骤**：
1. 设计无标题卡片的视觉规范（阴影、圆角、hover 效果需要更明显）
2. 若需要保留标题用于 SEO/A11y，可将其置于 aria-label 或 data 属性中
3. 实现 Hover 时的 tooltip 显示游戏名称（提升无障碍访问）

#### 9.3.2 网格布局参数

**建议采用以下网格参数**（参考 Poki 但根据实际情况调整）：

```css
.game-grid {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(94px, 1fr));
  gap: 16px;
}
```

- 使用 `auto-fill` + `minmax` 替代固定列数，可以自然实现响应式
- 卡片尺寸 94×94px 在高分辨率屏幕上可能需要调大，建议建立 94/120/150 三种尺寸规格

#### 9.3.3 卡片阴影与 Hover

```css
.game-card {
  border-radius: 16px;
  box-shadow: rgba(93, 107, 132, 0.3) 0px 7px 10px 4px;
  transition: transform 0.6s cubic-bezier(0.25, 0.1, 0.25, 1);
  cursor: pointer;
}

.game-card:hover {
  transform: scale(1.05);
}
```

### 9.4 首页结构优化

#### 9.4.1 首屏设计

**建议在首页顶部增加 Teal/Mint 薄荷绿首屏区块**：
- 可放置平台特色游戏推荐（3-5 款焦点游戏大图）
- 或放置品牌标语/价值主张
- 视觉上立即传达品牌调性

#### 9.4.2 内容分层

**建议将首页内容按以下层次组织**：

```
首屏区块（薄荷绿）→ 游戏网格（162+款）→ 分类卡片区 → 页脚
```

移除传统侧边栏，实现全宽内容流。

#### 9.4.3 页脚波浪过渡

**建议在内容区和页脚之间增加 SVG 波浪过渡**：
- 使用薄荷绿或渐变色波浪 SVG
- 高度建议 50-80px
- 可使用 CSS `clip-path` 或 SVG `<path>` 实现

### 9.5 游戏详情页优化

#### 9.5.1 游戏框架布局

**建议将游戏详情页的游戏框架调整为 16:9 比例**（参考 Poki 836×470），居中展示，上方留有适当空间显示游戏信息。

#### 9.5.2 游戏信息精简

**建议采用与 Poki 相似的精简信息布局**：
- 游戏标题（H1）
- 开发者名称（"by XXX" 格式）
- 点赞/点踩按钮（图标 + 数字）
- 全屏按钮、举报按钮

#### 9.5.3 推荐系统

**建议在详情页底部增加推荐游戏网格**：
- 数量：60-100 款（可配置）
- 使用与首页相同的卡片样式
- 可按同类游戏或同开发者聚合

### 9.6 搜索体验升级

#### 9.6.1 全屏搜索覆盖层

**建议实现以下搜索流程**：
1. 点击搜索图标 → 全屏半透明暗色遮罩展开
2. 搜索框自动获得焦点
3. 输入时实时显示搜索建议/结果
4. 按 ESC 或点击遮罩外部关闭

#### 9.6.2 搜索结果展示

搜索结果建议使用与首页相同的游戏卡片格式，保持视觉一致性。搜索关键词高亮显示。

### 9.7 广告策略建议

#### 9.7.1 Ad-Light 原则

**建议采用与 Poki 相似的广告策略**：
- 首页完全不显示广告 banner
- 广告仅在游戏加载时展示（插屏式）
- 游戏过程中不插播广告（除非是游戏内购买提示）

#### 9.7.2 广告文案设计

若需要展示"游戏即将开始"的等待提示，建议使用友善文案（如 "游戏即将开始，请稍候..."），避免使用"广告"字样。

### 9.8 移动端适配

#### 9.8.1 响应式断点

**建议建立以下响应式断点**：

```css
/* 手机 */
@media (max-width: 767px) {
  .game-grid { grid-template-columns: repeat(2, 1fr); gap: 12px; }
  .game-card { border-radius: 12px; }
}

/* 平板 */
@media (min-width: 768px) and (max-width: 1023px) {
  .game-grid { grid-template-columns: repeat(4, 1fr); gap: 14px; }
}

/* 桌面 */
@media (min-width: 1024px) {
  .game-grid { grid-template-columns: repeat(6, 94px); gap: 16px; }
}
```

#### 9.8.2 移动端导航

移动端建议将浮动导航缩小并固定在左上角，或改为顶部横条（包含 Logo + 搜索图标 + 汉堡菜单）。

### 9.9 性能优化

#### 9.9.1 图片 CDN

**建议建立图片 CDN 策略**，支持多种尺寸规格：

```
https://img.gamezipper.com/cdn-cgi/image/q=78,scq=50,width=94,height=94,fit=cover,f=auto/{game-id}/{game-name}.png
https://img.gamezipper.com/cdn-cgi/image/q=78,scq=50,width=204,height=204,fit=cover,f=auto/{game-id}/{game-name}.png
```

#### 9.9.2 资源压缩

Poki 首页 HTML 仅 385KB，建议 GameZipper 将首页 HTML 目标控制在 500KB 以内。

#### 9.9.3 懒加载

游戏网格中非首屏的卡片建议使用懒加载（Intersection Observer），减少首屏加载时间。

### 9.10 A/B 测试建议

在实施上述设计建议前，建议针对以下关键决策进行 A/B 测试：
