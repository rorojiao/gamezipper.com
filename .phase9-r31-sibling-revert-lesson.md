# R31 Lesson — Sibling Repeatedly Reverted My Registration During Build (2026-07-03)

## Scenario

Round 31, Quoridor (#531). I delegated the full pipeline to an orchestrator
subagent (Pattern C) which built `quoridor/index.html` + assets + registration
edits. A sibling cron (ops-gamezipper) was running CONCURRENTLY and chose a
DIFFERENT game (Nerdle) for the same #530 slot.

The collision sequence:
1. My subagent built Quoridor files + started registration (games-data.js,
   itemlist-schema.js, game-footer.js, sitemap.xml, index.html all modified).
2. Sibling committed `feat: add Nerdle #530` (commit 6b4bb18144) — its
   `git add -A` swept the working tree, but my unstaged edits to games-data.js
   + index.html were ALSO in the working tree, so they got included.
3. Sibling then committed a DAILY ANALYSIS commit (406679c872) which RESET
   games-data.js + index.html to the pre-Quoridor state (the daily analysis
   script regenerates those files from a canonical source, dropping my manual
   Quoridor entry).
4. Sibling then committed `fix: remove orphan Quoridor entries, restore
   4-source sync to 530` (6f50dd65dc) — it detected my partial registration
   as "orphan entries" and REVERTED them.

My registration was overwritten 3 times before I caught it. Each time I
re-added the entry, the next sibling commit removed it.

## Signature

- Your orchestrator returns "registration complete."
- `node -e "...GAMES.length"` shows your slug, but `git status` shows
  games-data.js and index.html as CLEAN (not modified) — because a sibling
  commit already reverted them.
- `git log --oneline -5` shows sibling-authored commits at HEAD, including
  a `fix: remove orphan <your-slug> entries` or `chore: update pipeline
  status - Round N (<other-slug> #N) complete`.
- Your `quoridor/` directory IS on disk and committed, but `grep -c quoridor
  js/games-data.js` returns 0.

## Correct response

1. **Recognize the slot is taken.** The sibling's game is now #N. Your game
   must be #N+1. Re-number your entry: position N+1 in itemlist-schema.js,
   numberOfItems N+1, all count references N→N+1.
2. **Re-add your entry AND immediately commit in one atomic shell.** Do NOT
   use the patch tool or separate write+commit steps — the ~10s gap between
   "write file" and "git add" is enough for another sibling commit to revert.
   Use:
   ```bash
   cd /home/msdn/gamezipper.com
   python3 << 'EOF'
   # 1. games-data.js: insert entry #N+1
   # 2. itemlist-schema.js: append ListItem position N+1, numberOfItems N+1
   # 3. game-footer.js: append entry
   # 4. sitemap.xml: append <url>
   # 5. index.html: regex-replace all count refs N→N+1 + cat-count puzzle +1
   # ALL in one python script
   EOF
   git add js/games-data.js js/itemlist-schema.js game-footer.js sitemap.xml index.html og-images/<slug>.png <slug>/
   git commit -m "feat: add <Name> (#N+1) ..."
   ```
3. **Verify immediately:** `bash scripts/sync-game-counts.sh` must show
   "ALL IN SYNC" at N+1 before you walk away.
4. If the pre-commit hook fails, fix and re-commit in the SAME terminal
   invocation (don't leave the worktree dirty).

## Prevention

**The daily-analysis cron is the silent killer.** It runs as a sibling, reads
games-data.js + index.html as canonical inputs, regenerates derived files, and
commits — dropping any manual edits that aren't in its source. If your
registration sits unstaged when the daily-analysis cron fires, it's gone.

**Two mitigations:**
1. **Commit registration in a single atomic shell** (python writes + git add
   + git commit in one bash invocation). The R63 Pattern-A advice applies
   doubly here.
2. **After any sibling-race round, re-verify HEAD within 60s of your commit.**
   Run `git show HEAD:js/games-data.js | grep -c <slug>` — if it's 0, the
   sibling reverted you again; re-apply.

## Distinction from R30 and R63

- **R30**: sibling committed MY game (ships-finder) in ITS commit, correctly.
  Single collision, clean resolution (just verify, don't re-register).
- **R63 Pattern A**: sibling's `git add -A` split my registration across two
  commits. Harmless once both are in HEAD.
- **R63 Pattern B**: sibling left an orphan `<slug>/` dir unregistered.
- **R31 (THIS)**: sibling actively REVERTED my registration 3× because it
  (a) took the #530 slot with its own game, (b) ran a daily-analysis regen
  that dropped my edits, (c) explicitly "fixed" my entries as orphans. The
  fix is re-numbering to #N+1 + atomic commit + 60s re-verify.

## Also: check sibling's game for policy violations

The sibling's Nerdle (#530) shipped with a 1ktower zombie-endpoint `<script>`
reference (R188 violation — the pre-commit hook should have caught it but the
sibling may have bypassed or the hook was added after). Always run
`grep -rn 1ktower <sibling-slug>/` on any sibling-shipped game and fix it in
your follow-up commit. The R188 policy applies to ALL games, not just yours.
