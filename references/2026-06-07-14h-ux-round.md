The CSS rule at L369 `.faq-item.open .faq-answer{max-height:200px;opacity:1;margin-top:8px}` is **orphaned** — no JS path adds the `.open` class to `.faq-item`. Instead, the JS at L2004-2019 sets `p.style.display='none'` on the inner `<p>` (a totally different element from `.faq-answer`). The result: ALL 9 FAQ answers are permanently hidden (max-height:0; opacity:0; overflow:hidden), and clicking the question heading does nothing visible.

### Why the bug went undetected
- **Visual deception**: The HTML structure has `<div class="faq-item"><h3>...</h3><div class="faq-answer"><p>...</p></div></div>`. The JS correctly identified "the answer is inside a `<p>`" (common accordion pattern), so the inline `p.style.display='none'` seemed to do something.
- **CSS test passed**: The CSS `.faq-item .faq-answer{max-height:0;...}` DID hide the answer (so visually the FAQ LOOKED correct at page load — all answers collapsed).
- **But click test was never run in QA**: The click handler toggled `<p>` display:none/block, but the wrapping `.faq-answer` div is still `max-height:0; overflow:hidden`, so the inner `<p>`'s display change had **no visible effect**.
- **10h round's interpolate-size activation did NOT catch this**: That round added `interpolate-size:allow-keywords` to `.faq-answer`, expecting the `.faq-item.open` class to be toggled. It was never toggled, so the new CSS (and the old max-height:200px) never fired. The 10h round was building on a broken foundation.

### Detection recipe
1. Find the toggle handler: `grep -n "addEventListener.*click.*toggle\|toggle()" index.html | head -10`
2. Find the CSS selector: `grep -nE "\.faq-item[\\. ]|\.faq-answer" index.html | head -20`
3. **Check that the JS toggled element MATCHES the CSS targeted element**. If JS toggles `p.style.display` but CSS targets `.faq-answer` — **MISMATCH BUG**.
4. Live Kachilu test: `document.querySelector('.faq-item .faq-answer').getBoundingClientRect().height` — should be 0 (collapsed) initially, then >0 after click. If it stays 0 forever, the bug is present.

## Hard Rule 32 (NEW) — Toggle handlers must target the same element the CSS targets
Whenever you have a `classList.toggle('open')` pattern (or similar) in JS combined with `.parent.open .child{}` CSS, **the JS toggled element MUST be the same element the CSS descendant selector targets**. If the JS toggles `p.style.display` but the CSS rule is `.faq-item.open .faq-answer`, the rule never fires and the entire feature is silently broken. **Detection recipe**: `grep -nE "(classList\\.toggle|add\\(['\"]|remove\\(['\"])" index.html | head -20` to find all toggles, then for each one `grep -n "<the_class>" index.html | head -10` to find CSS targets, and **verify they match**. Even better: write a tiny test snippet that toggles the class on the correct element and checks `getComputedStyle(child).maxHeight !== '0px'` — fails fast on the live page.

## Future migration candidates (per Hard Rule #30 spirit)
- All `let hidden = isHidden(elem)` patterns followed by `hidden ? elem.style.display='none' : elem.style.display='block'` are vulnerable to this bug class. Audit: `grep -nE "style\\.display=.*\?.*'none'|style\\.display=.*'block'" index.html | head -10`
