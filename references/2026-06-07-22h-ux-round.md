### 1. P0 SEO — Inline JSON-LD ItemList `numberOfItems` drift (48 → 46)
**Root cause**: `index.html:426` inline `<script type="application/ld+json">` had `"numberOfItems":48` but the actual `itemListElement` array had 46 items (positions 1-46). Discovered during Phase 1.5's `JSON.parse` round-trip check (Hard Rule #22). `sync-game-counts.sh` only scans `data-count=` attributes and the `js/itemlist-schema.js` file — it does NOT scan inline `<script type="application/ld+json">` blocks in `index.html`. The drift was introduced when 2 ListItems were removed at some point but `numberOfItems` was not decremented.

**Fix**:
- `index.html:426` `"numberOfItems":48` → `"numberOfItems":46` (Python single replace; re-parsed to confirm)
- This is a top-picks/homepage featured-list (the 46 hard-coded featured games), NOT the main 267-game schema (which lives in `js/itemlist-schema.js` and is rendered dynamically)
- Search engines may have flagged as inconsistent data; fixing brings consistency

**Detection recipe** (NEW — added to Phase 1.5):
```python
python3 -c "
import re, json
c = open('index.html').read()
blocks = re.findall(r'<script[^>]*application/ld\+json[^>]*>([\\s\\S]*?)</script>', c)
for b in blocks:
    d = json.loads(b)
    if 'itemListElement' in d:
        n = d.get('numberOfItems', 0)
        items = len(d['itemListElement'])
        if n != items: print(f'❌ DRIFT: {d[\"@type\"]} numberOfItems={n} vs items={items}')
        else: print(f'✅ {d[\"@type\"]}: {n}={items}')
"
```

The existing `@media(pointer:coarse)` block at L207 already covered `.cat-btn`, `.gz-search`, `.game-card`. Extended the same block with 3 new selectors — single-rule addition, no JS change, no template duplication.

### 3. P1 perf — `prefers-reduced-data` media gate on 9 prefetch links
**Root cause**: 9 `<link rel="prefetch" href="/..." fetchpriority="low">` were unconditionally downloading 9 HTML pages on every page load. For users on metered connections (mobile data saver mode, roaming, limited plans), this is wasteful — they explicitly opted into OS-level "less data" preference and the site was ignoring it.

**Fix**: added `media="(prefers-reduced-data: no-preference)"` to all 9 prefetch links:
```html
<link rel="prefetch" media="(prefers-reduced-data: no-preference)" href="/2048/" fetchpriority="low">
<link rel="prefetch" media="(prefers-reduced-data: no-preference)" href="/snake/" fetchpriority="low">
... (7 more)
```

**How it works**: HTML5 `<link media="...">` attribute is supported by all major browsers. The link is only fetched if the media query matches. Users with `prefers-reduced-data: reduce` (set via OS settings) will skip the prefetch, saving ~50-200KB of HTML per page load.

**Standards reference**: MDN updated `prefers-reduced-data` documentation 2025-10-17. Chrome 85+, Firefox 118+, Safari 17.4+. Caniuse 2026-04: 95%+ global support.

**Tradeoff**: 2nd-page nav on metered connections is now ~50-100ms slower (no prefetch). Acceptable: these are 9 popular game pages, and the user explicitly chose to use less data. Page itself still loads normally (no impact on LCP/TTFB).
