## Key Learning
Hard Rule #28 (08h round) — but extended: when a 2026 CSS feature is declared on a high-level ancestor (`:root`, `body`, etc.) as a "future migration" enabler, the actual descendant migration MUST be a follow-up round. The 20h round noted "unlocks future migration to height:0/auto without changing current behavior" but never did it. Setting the feature flag is half the work — the migration IS the other half.

With Chrome 129+ coverage at ~85% in 2026, `height:0/auto` transitions are universally safe. The 200px max-height cap on `.faq-answer` was clipping long answers (e.g., "How often are new games added?" is 188 chars and barely fits). Now supporting browsers get smooth, content-sized accordion expansion; older browsers get the original 200px behavior.

## Hard Rule #28 Extended (Proposed)
"After declaring any 2026 CSS feature on a high-level ancestor as 'future migration' enabler, the descendant migration should be tracked as a TO-DO and completed within 2-3 rounds. Audit recipe: `grep -n 'interpolate-size:allow-keywords' index.html` — should appear in BOTH ancestor (set flag) AND descendant (use it) contexts. Single occurrence = dormancy."
