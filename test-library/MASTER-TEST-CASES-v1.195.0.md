## New Test Cases Added in v1.195.0 (R312 — July 16 2026, 4h cycle 74)

### Security (S)

- [S-437] **[P0]** CVE-2026-7333 Chrome GPU Process UAF Sandbox Escape (CVSS 9.6, Apr 28 2026, Chrome 147.0.7727.138 Win/Mac + 147.0.7727.137 Linux) — Use-After-Free in Chrome's GPU Process Allows Complete Sandbox Escape via Malicious Web Page; Highest-Severity 2026 Chrome CVE Catalogued (CVSS 9.6 > CVSS 8.x of S-435/CVE-2025-6558); "Critical" Tier in Google Advisory; GPU Process Has Higher Privilege Than Renderer Sandbox (blog.csdn.net/weixin_42376192/article/details/160846721 May 7 2026 — "2026年4月28日,Google紧急推送Chrome浏览器稳定版更新,修复了一个编号为CVE-2026-7333的高危GPU进程释放后重用(UAF)漏洞｡该漏洞CVSS 3.1评分高达9.6分,可被恶意网页利用实现完整沙箱逃逸,进而完全接管用户系统...该漏洞并非存在于传统的渲染器进程,而是位于权限更高的GPU进程中"). **Why this is the GPU-PROCESS-UAF-SANDBOX-ESCAPE-CVE-2026-7333 dimension distinct from S-435 (CVE-2025-6558, TAG-discovered Chrome 0-day, renderer-side) and B-220 (CVE-2025-9478, ANGLE UAF)**: S-435 covered a renderer-process V8/Dawn 0-day patched Jul 15 2025. B-220 covered an ANGLE (WebGL backend) UAF discovered by Google's Big Sleep AI agent (Aug 2025). S-437 captures the **GPU-PROCESS-PRIVILEGE-ESCALATION** dimension — CVE-2026-7333 lives in the GPU process, which has HIGHER OS privileges than the renderer (it directly accesses GPU drivers, shared memory, and graphics APIs that the renderer cannot touch). A GPU-process UAF therefore escapes the renderer sandbox AND bypasses site-isolation boundaries that protect renderer-process bugs. Combined with CVSS 9.6 (the highest in the 2026 Chrome CVE line catalogued so far), this is the **most severe browser-side exploit path to GameZipper users in 2026**. **Why this matters for GameZipper**: (a) every WebGL GameZipper game (and every 2D Canvas game that uses GPU compositing, which is nearly all of them via `willReadFrequently:false` default) routes drawing commands through the GPU process — a malicious ad creative loaded in monetag-manager.js's iframe can directly trigger the GPU-process code path, making GameZipper's ad surface the delivery vector; (b) sandbox escape means the attacker escapes Chrome entirely and runs as the user — GameZipper cannot prevent this game-side but MUST record incident-correlation data (Chrome full patch level, GPU-process crash signatures) so post-incident forensics can attribute a breach; (c) bundled-Chromium browsers (360/QQ/Sogou per W-372, confirmed ≥1 major behind Chrome stable) may ship Chromium 146 or earlier, leaving their users permanently exposed to CVE-2026-7333 even months after the patch; (d) the CVSS 9.6 score + GPU-process location + full-sandbox-escape impact = this CVE should be treated as a supply-chain-incident-trigger threshold: any spike in GPU-process crashes in GameZipper telemetry after an ad-campaign change is a possible CVE-2026-7333 exploitation signal. **Detection**: static — verify GameZipper records `navigator.userAgentData.brands` AND `chrome.runtime.getPlatformInfo()` (or equivalent) for full patch-level capture; dynamic — on Chrome <147.0.7727.138, run a GPU-process stress probe (WebGL drawArrays loop with deliberately malformed vertex attributes) and check for GPU-process crash (`chrome://gpu` shows "GpuProcess crashed" status); telemetry — instrument `webglcontextcreationerror` + `CONTEXT_LOST` events as a GPU-process-instability signal.

- [S-438] **[P1]** CVE-2026-40369 Windows 11 Kernel Sandbox Escape via Browser-Mediated NtQuerySystemInformation (CVSS 7.8, May 21 2026, Win11 24H2/25H2) — UAF in `ntoskrnl.exe!ExpGetProcessInformation`; Attacker in Chrome Sandbox Calls `NtQuerySystemInformation(InfoClass 253)` to Bypass `ProbeForWrite` and Achieve Deterministic SYSTEM-Level Privilege Escalation; Originally Prepared for Pwn2Own Berlin 2026 (Disclosed Early by Researcher Paolo Stagno); Trigger Requires Browser-Process Access to Win32 API (www.donews.com/news/detail/8/6567089.html May 22 2026 — "2026年5月21日,科技媒体cyberkendra披露Windows 11内核存在严重漏洞CVE-2026-40369(CVSS 7.8),影响24H2及25H2版本｡该漏洞位于ntoskrnl.exe的nt!ExpGetProcessInformation函数,攻击者可通过Chrome等浏览器调用NtQuerySystemInformation(信息类253)触发,绕过ProbeForWrite校验,实现从沙箱到系统最高权限的确定性提权｡漏洞由研究者Paolo Stagno发现,原计划用于Pwn2Own Berlin 2026,后因提前公开致参赛中止"). **Why this is the OS-LAYER-SANDBOX-ESCAPE-VIA-BROWSER dimension distinct from S-437 (GPU-process UAF, in-browser) and S-435 (renderer V8 0-day, in-browser)**: S-437 and S-435 are in-browser exploits that compromise Chrome's own processes. S-438 captures the **OS-KERNEL-COMPOUND-CHAIN** dimension — the bug is in the Windows kernel (`nt!ExpGetProcessInformation`), but it is **triggered from inside the browser sandbox** via `NtQuerySystemInformation`. This means a GameZipper ad-side initial-access (e.g., S-437 GPU-process escape) can be chained with S-438 to escalate from sandboxed-renderer → GPU-process → Windows SYSTEM. The Pwn2Own Berlin 2026 attribution confirms this is a **competition-grade reliable exploit**, not a theoretical bug — meaning the compound chain is reproducible by skilled attackers. **Why this matters for GameZipper**: (a) the Windows 11 24H2/25H2 install base is the modal desktop OS for GameZipper's traffic (per W-373 Chrome 150 baseline) — a large fraction of users are running an OS version with an unpatched kernel UAF; (b) GameZipper cannot patch the OS, but should DETECT the threat surface: any QA verdict on Win11 24H2/25H2 + Chrome pre-fix should be flagged as a compound-chain-exposed session; (c) the browser-mediated NtQuerySystemInformation call path means GameZipper's ads/trackers/analytics could theoretically be the initial-access vector — GameZipper should audit third-party scripts for any `NtQuerySystemInformation`-equivalent Win32 API surface (rare in JS, but possible via NativeMessaging / PPAPI / WebAssembly-with-syscalls); (d) the Pwn2Own origin means the exploit may already be in circulation among nation-state/criminal crews even if no public ITW confirmation exists yet. **Detection**: static — verify GameZipper records Windows build number (`navigator.userAgentData.getHighEntropyValues(['platformVersion','architecture','bitness','wow64'])`) in every verdict; dynamic — on Win11 24H2/25H2 + unpatched kernel, the only reliable detection is OS telemetry (Event Log ID 4720/4624 anomaly); GameZipper-side: flag any session where renderer-process crashes co-occur with SYSTEM-level privilege events as a possible S-438 chain.

- [S-439] **[P2]** CVE-2025-0999 Chrome V8 Heap Buffer Overflow (Feb 2025, Patched in Chrome 133.0.6943.126 Win/Mac + 133.0.6943.126 Linux) — V8 Heap Buffer Overflow Allows Arbitrary Code Execution from a Malicious Web Page; No User Authentication Required; Trigger is Remote (HTTP page load); Threat Level: High (fjpit.edu.cn/xxzx/2025/0507/c2958a125774/page.htm May 7 2025 — "Chrome 浏览器 JavaScript 引擎 V8 中存在一个严重的堆缓冲区溢出漏洞,攻击者可以利用该漏洞执行任意代码并获取用户系统权限导致失陷...影响范围: Google Chrome < 133.0.6943.126 ...综合评定利用难度:困难,能执行任意代码...综合评定威胁等级:高危,能导致系统失陷"). **Why this is the V8-HEAP-BUFFER-OVERFLOW dimension distinct from S-436 (CVE-2025-6554 V8 optional-chaining type-confusion Hole Check) and B-221 (CVE-2025-10585 `Symbol.toPrimitive` type-confusion)**: S-436 and B-221 are V8 type-confusion bugs (the interpreter mis-classifies an object's type and reads the wrong field). S-439 captures the **V8-HEAP-MEMORY-CORRUPTION** dimension — a classic heap buffer overflow where V8 allocates a buffer too small for the data written into it, corrupting adjacent heap objects. This is a different exploitation primitive (write-what-where vs type confusion) and historically produces more reliable exploits (heap overflows have decades of exploitation tooling). **Why this matters for GameZipper**: (a) the patched floor is Chrome 133 (Feb 2025) — bundled-Chromium browsers (W-372) and Electron apps bundling Chromium <133 remain vulnerable 17+ months after the patch; (b) GameZipper's third-party ad iframes can deliver a crafted HTML page that triggers the V8 overflow — same delivery vector as S-435/S-436/S-437; (c) the "high threat level, can cause system compromise" rating means this is in the same severity tier as S-435; the lower P2 (vs S-435's P0) reflects only the older patch date and the absence of confirmed in-the-wild exploitation, not lower technical severity; (d) useful as a regression-baseline for V8 hardening — if a future Chrome V8 change re-introduces a heap-overflow class, GameZipper's incident-correlation logs should be able to identify sessions running pre-133 versions as the likely vulnerable population. **Detection**: static — verify GameZipper captures Chrome full version (not just major) in every verdict; dynamic — on Chrome <133.0.6943.126, run a V8 heap-stress probe (large TypedArray allocations + setReturnType coercion) and check for renderer crash.

### Web Platform / Browser (W)

- [W-375] **[P1]** Chrome 154 "Always Use Secure Connections" Default-On (Effective Oct 2026, Announced Oct 28 2025) — From Chrome 154, Browser Will DEFAULT to Warning Full-Page Interstitial for ANY HTTP Public Website; User Must Click-Through to Proceed; Per-Site Choice Remembered for Frequently-Visited HTTP Sites; Optional Coverage of Private/Ethernet Sites (so.html5.qq.com/page/real/search_news?docid=70000021_163690194b743552 Oct 29 2025 — "从 2026 年 10 月发布的 Chrome 154 版本开始,谷歌将默认启用'始终使用安全连接'(Always Use Secure Connections)功能｡这项变更意味着,用户届时尝试访问未使用 HTTPS 加密协议的公共网站时,浏览器将不再直接连接,而是会先弹出一个警告窗口,请求用户确认后才能继续访问...对于用户经常访问的 HTTP 站点,浏览器会记住用户的选择,不再重复警告"). **Why this is the HTTPS-ONLY-WARNING-DEFAULT-ON-CHROME-154 dimension distinct from W-373 (Chrome 150 stable floor) and the existing Phase 0.5 mixed-active-content detector**: W-373 covers version-floor baselines. The Phase 0.5 detector catches `<script src="http://">` (mixed active content). W-375 captures the **USER-FACING-WARNING-UX** dimension — Chrome 154 will show a full-page red interstitial for ANY outbound navigation to HTTP, including `<a href="http://...">` link clicks, `<img src="http://...">` (image warnings are softer but present), `<iframe src="http://...">` (blocked), and `window.open('http://...')` calls. GameZipper itself is HTTPS so the homepage won't trip the warning, BUT: (a) any GameZipper outbound link to an HTTP analytics/ad/CDN endpoint will trigger the warning when a user clicks it, severely damaging UX and ad-revenue flow; (b) any GameZipper ad-creative served via an HTTP iframe will be silently blocked by Chrome 154, breaking monetization invisibly (the ad container stays empty, no console warning); (c) the per-site-remembered exception is per-user-per-device, meaning GameZipper's analytics cohort on Chrome 154+ will fragment (some users click-through, some don't) — affecting data integrity. **Why this matters for GameZipper**: (a) gamezipper.com is HTTPS so the warning won't appear on the homepage, but EVERY outbound HTTP reference in GameZipper HTML/JS becomes a warning trigger; (b) the static-sweep detector in `post-deploy-verification` Phase 0.5 currently catches only `<script src="http://">` — it MUST be extended to all HTML attributes and JS APIs that Chrome 154 will warn on (see PC-010 for the concrete sweep); (c) ad networks (monetag-manager.js, Adsterra, AdSense) frequently serve creative over HTTP despite the parent page being HTTPS — Chrome 154 will block these silently, and GameZipper's ad-revenue will drop without any visible error; (d) the Oct 2026 enforcement date is ~3 months away from this cycle (Jul 16 2026) — GameZipper has a hard deadline to audit and migrate all HTTP references. **Detection**: static — run the PC-010 extended HTTP-reference sweep; dynamic — on Chrome 154+, navigate to a GameZipper page with ad creative and check whether the ad container fills (if empty + no console error, suspect HTTP-creative blocking by Always-Use-Secure-Connections).

- [W-376] **[P2]** 2026 H1 CVE Volume Leaderboard — Chrome 1584 CVEs in First Half of 2026 (Linux 2308 #1 by Vendor, Google 1752 #2 by Vendor, Chrome 1584 #2 by Product); Patched-Floor Assumption Is Now Provably Fragile (so.html5.qq.com/page/real/search_news?docid=70000021_1486a57636a99052 Jul 15 2026 — "按厂商排序,Linux以2308个CVE位居榜首,谷歌以1752个位列第二,微软843个排第四｡按产品排序,Linux仍以2309个居首,Chrome以1584个排第二" + new.qq.com/rain/a/20260704A02ZDW00 Jul 4 2026 — "2026 年上半年披露的 CVE 漏洞排名中,Linux 内核以 2309 条位居榜首,谷歌以 1752 个位居第二"). **Why this is the CVE-VOLUME-PATCHED-FLOOR-FRAGILITY meta-pattern distinct from W-373 (Chrome 150 floor) and W-374 (rapid release cadence)**: W-373 establishes a single version floor. W-374 captures the release-cadence compression. W-376 captures the **VOLUME-DRIVEN-FRAGILITY** dimension — at 1584 Chrome CVEs in 6 months (~265/month, ~9/day), the assumption that "Chrome version N is patched" is provably wrong: between any two QA ticks (4h apart), an average of ~1.5 new Chrome CVEs are disclosed. This means: (a) a verdict recorded at tick T saying "Chrome 150 = patched against S-437" is stale by tick T+1 if a new GPU-process CVE drops in the interim; (b) GameZipper's incident-correlation logs MUST record Chrome FULL version + build + days-since-release, not just major version, to be useful for forensic attribution; (c) the CVE volume validates the 4h-cadence Dynamic Test Intelligence cron — at 9 CVEs/day, a daily or weekly review cadence misses ~75% of new threats; the 4h cadence (6 ticks/day) captures ~75% of new CVEs within their first disclosure window. **Why this matters for GameZipper**: (a) the existing S-category test cases (S-435, S-436, S-437, S-438, S-439) each reference specific CVEs — W-376 establishes that these are a SAMPLE of a much larger population, and the threat model should assume N new high-severity Chrome CVEs per month; (b) QA verdicts recording only `navigator.userAgent` (which gives Chrome major version) are insufficient — GameZipper should capture `navigator.userAgentData.getHighEntropyValues(['fullVersionList','platformVersion'])` to record the full Chrome patch level; (c) the Linux 2308 CVE leaderboard is relevant to GameZipper's backend infrastructure (if any server-side rendering or headless-browser QA runs on Linux, the Linux CVE volume is the backend threat surface). **Detection**: meta-metric — track the count of new Chrome CVEs disclosed since the last QA tick; if >0 high-severity Chrome CVEs dropped since last tick, flag the current tick's verdicts as "CVE-stale" and re-baseline.

### Platform Compatibility (PC)

- [PC-010] **[P3]** GameZipper Outbound HTTP Reference Audit for Chrome 154 "Always Use Secure Connections" — Comprehensive Static Sweep for ALL `http://` References in HTML Attributes (`href`/`src`/`srcset`/`action`/`data`/`formaction`/`form`/`poster`/`background`/`cite`/`manifest`/`longdesc`/`usemap`/`profile`/`codebase`/`archive`/`icon`) and JS APIs (`WebSocket('ws://')` / `EventSource('http://')` / `navigator.sendBeacon('http://')` / `window.open('http://')` / `window.location='http://'` / `fetch('http://')` / `XMLHttpRequest.open('http://')` / `<meta http-equiv="refresh" content="0;url=http://...">`). **Why this is the EXTENDED-HTTP-REFERENCE-SWEEP-FOR-CHROME-154 dimension distinct from the existing Phase 0.5 mixed-active-content detector and PC-009 (V8 optional-chaining)**: The existing Phase 0.5 detector catches only `<script src="http://">` (the most severe mixed-active-content case, which Chrome already blocks). PC-009 covers V8 optional-chaining. PC-010 captures the **COMPREHENSIVE-OUTBOUND-HTTP-AUDIT** dimension required by Chrome 154's default-on Always-Use-Secure-Connections — Chrome 154 will warn/block on outbound HTTP regardless of whether the resource is "active" (script) or "passive" (image/font/CSS), extending the existing detector's scope by ~15 attribute types and ~7 JS APIs. **Why this matters for GameZipper**: (a) the existing detector returns 0 hits on gamezipper.com for `<script src="http://">` (already migrated to HTTPS), BUT has never audited the broader attribute/API surface that Chrome 154 will warn on; (b) third-party ad scripts (monetag-manager.js, Adsterra, AdSense) frequently inject HTTP image/font/iframe resources at runtime — these will be silently blocked by Chrome 154, causing ad-revenue drop without any console error; (c) the Oct 2026 Chrome 154 enforcement date gives GameZipper a hard 3-month deadline (from this Jul 16 2026 cycle) to complete the audit; (d) the sweep should run as a pre-commit hook to prevent new HTTP references from entering the codebase. **Detection**: static — extend the Phase 0.5 detector per the W-375 attribute/API list; commit-time — block any PR that introduces a new `http://` reference outside an explicit allowlist (e.g., `schema.org`, `og:image`); runtime — on Chrome 154+, instrument `SecurityPolicyViolation` events and `block-mixed-content` console warnings to catch runtime-injected HTTP resources that static-sweep misses.
