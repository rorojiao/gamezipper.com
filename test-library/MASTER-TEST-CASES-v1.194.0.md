- [S-436] **[P1]** CVE-2025-6554 Chrome V8 Type-Confusion via Optional-Chaining "Hole Check" Flaw (CVSS 8.1, Sep 3 2025) — V8 Interpreter "Hole" Verification Mechanism Fails on Optional-Chaining (`?.`) Expressions Over Uninitialized Variables, Allowing Reads of Non-Initialized Values Normally Blocked by Reference Checks; Affects ALL Chromium-Derived Browsers (Chrome, Edge, Brave, Opera, Electron Apps); PoC Published (github.com/ghostn4444/POC-CVE-2025-6554 + blog.csdn.net/weixin_38526314/article/details/150982391 Sep 3 2025 — "漏洞编号 CVE-2025-6554(CVSS 8.1),影响所有基于 Chromium 的浏览器(Chrome､Edge､Brave､Opera､Electron 应用等)" + github.com/ghostn4444/POC-CVE-2025-6554 — "CVE-2025-6554 is a vulnerability in Chrome's V8 (Engine JavaScript) interpreter (Hole Check) inadequate control in unintended variables within expressions with optional chaining. POC explores a flaw in the 'Hole' verification mechanism in variables before use, allowing non-initialized values reading-behavior that is normally blocked by a reference. `function leak_hole(){ let x; delete x?.[y]?.a; return y; let y; }`"). **Why this is the V8-OPTIONAL-CHAINING-HOLE-CHECK-CVE-2025-6554 dimension distinct from B-221 (CVE-2025-10585 `Symbol.toPrimitive`, v1.193.0) and S-434 (CVE-2025-13223 undocumented)**: B-221 covered a V8 type-confusion triggered by `Symbol.toPrimitive` handlers returning arrays. S-434 is undocumented. S-436 captures the **OPTIONAL-CHAINING-UNINITIALIZED-READ** dimension — the bug is in V8's "Hole Check" (the mechanism that prevents reading uninitialized `let`/`const` variables), specifically when optional chaining (`?.`) is applied to a variable that is later-declared (`let y` after the `?.[y]` access). This is a DIFFERENT V8 subsystem (interpreter Hole Check vs B-221's type-coercion) AND a different trigger pattern (`?.` on TDZ variables vs `Symbol.toPrimitive` proxies). **Why this matters for GameZipper**: (a) modern game code (post-2024) increasingly uses optional chaining (`state?.player?.score?.toString()`) — if a game has a `?.` chain that touches a TDZ variable (e.g., a `let` declared later in the same scope due to a refactor bug), the game may READ garbage/uninitialized memory on unpatched Chromium instead of throwing ReferenceError; this produces **non-deterministic game-state corruption** (random scores, item duplication, save-data poisoning) that is extremely hard to reproduce in QA; (b) Electron-based game wrappers (Discord activity, Steam browser, embedded webviews) bundle stale Chromium and are confirmed vulnerable; (c) the PoC is PUBLIC (github.com/ghostn4444), meaning exploit code is available to any attacker; (d) QA should flag any game using `?.` chains on `let`/`const` variables that are declared later in their scope (TDZ-optional-chaining pattern) as a CVE-2025-6554-amplification risk. **Detection**: static — `grep -nE '\?\.\[[a-zA-Z_]' SLUG/index.html` (optional-chaining on bracket-access of a bare identifier) cross-referenced with `let <same-identifier>` appearing LATER in the file; dynamic — on unpatched Chromium, trigger the game's optional-chaining code paths under a V8 debugger and check for non-ReferenceError reads.
