No memory corruption is involved; the extension has the privilege to inject content_script into any page including the privileged Gemini Live page, and the policy check is bypassable. This is fundamentally different from the S-435/S-437/S-438 threat class because those are bugs in Chrome's own code, but S-440 is **abuse of Chrome's intentional design** for compromised-extension delivery. **Why this matters for GameZipper**: (a) any GameZipper player who has a compromised extension installed (e.g., from chrome-webstore-trust-gap surface, see W-378) can have their camera/mic/screenshots exfiltrated by that extension WHILE playing a GameZipper game — the extension sees all page content via `content_scripts` and can act on the user's behalf when they open Gemini Live (or any other privileged Chrome UI); (b) GameZipper cannot enumerate which of its users have compromised extensions, but should DETECT attempts by GameZipper-side code to call restricted APIs (camera/mic via getUserMedia, Geolocation, Clipboard) without user gesture — these are signals of compromised-extension activity in QA verdicts; (c) the WebView tag-policy-bypass angle means GameZipper games embedded in Android WebView via third-party apps are also affected — a malicious wrapper app could embed GameZipper and exfiltrate user interaction data; (d) for QA: in tests that record `navigator.mediaDevices.enumerateDevices()` to detect virtual-camera permission grants, the policy-bypass attack may grant camera without showing a permission prompt, so QA verdicts should explicitly check `permissions.query({name:'camera'}).state === 'granted'` AND a corresponding prompt was shown to the user; (e) the high-credibility phishing angle means GameZipper's own UI could be imitated by a malicious Gemini Live overlay, so verification pages should include a fingerprint check that asserts the page is gamezipper.com via DOM characteristics (e.g., `document.querySelector('meta[property="og:site_name"]')?.content === 'GameZipper'`) before performing sensitive actions.

S-442 captures the **EXTENSION-ECOSYSTEM-ATTACK-SURFACE** dimension — a code-bug in a third-party Chrome extension (not in Chrome) allowed a compromised operator to push arbitrary code via that extension's context. This is the **supply-chain attack** on the extension ecosystem. GameZipper's users who have Adblock-style extensions installed are exposed because the extension's injected code runs with the extension's privileges (which often include `content_scripts` permission that lets them inject JavaScript into any page including GameZipper). **Why this matters for GameZipper**: (a) GameZipper's defense-in-depth model assumes Chrome itself is a trusted environment, but compromised extensions break that trust boundary by injecting JavaScript that can scrape GameZipper scores, hijack clicks (e.g., on the related-games footer links), exfiltrate localStorage game-saves, or replace ad containers to redirect monetization; (b) the "Adblock for YouTube" pattern is broadly replicated across ad-blocker / privacy / coupon / screenshot extensions — GameZipper's user base likely has ≥1 such extension active, and a single compromised extension can affect ALL of GameZipper's users simultaneously; (d) the **extension-CSP** enforcement is the technical mitigation here — GameZipper's CSP can be set to `default-src 'self'` to prevent any extension-injected scripts from running, but this requires GameZipper's HTML to declare CSP via `<meta http-equiv="Content-Security-Policy">` or response header, which most current GameZipper pages do NOT — this gap is a P1 backlog item; (e) the **server-side-config-injection** vector means that even GameZipper's own ad-script CDN (monetag-manager.js) could become a similar supply-chain compromise target if anyone in the supply chain is breached — defense requires SRI (`integrity="sha384-..."`) on all GameZipper-loaded scripts, which is a P1 backlog item already tracked in W-376's CVE-volume threat model.

### Game Performance (G)

- [G-195] **[P3]** INP (Interaction to Next Paint) Core Web Vital Threshold Check (<200ms "Good") — INP Replaced FID as the Core Web Vital Responsiveness Metric in May 2024; Targets Click/Tap/Key Press Events; Threshold: <200ms = "Good", 200-500ms = "Needs Improvement", >500ms = "Poor"; GameZipper Games Are Interaction-Heavy (Click, Drag, Tap, Key) Making INP the Most Relevant Field Performance Metric for the GameZipper Traffic Profile; Measured in Real User Monitoring (RUM) via Event Timing API (developer.mozilla.org/en-US/docs/Glossary/Interaction_to_next_paint Jan 14 2026 — "INP was designed by Google as one of the Core Web Vital metrics, replacing First Input Delay (FID) in May 2024"). **Why this is the INTERACTION-RESPONSIVENESS-FIELD-METRIC dimension distinct from existing G-category tests (which focus on canvas pixel-checks, button-render, persistence)**: G-195 captures the **FIELD-MEASURED-RESPONSIVENESS-METRIC** dimension — INP is the Core Web Vital metric that quantifies the user-perceived responsiveness of a page, measuring the worst interaction delay across the entire session. GameZipper games have many interactions per second (click-to-place-tile, drag-to-aim, tap-to-jump), so a single bad INP event is enough to drop the entire session's INP score into "Needs Improvement" or "Poor" territory. Google Search uses Core Web Vitals as a ranking signal — INP-poor pages rank lower, which directly affects GameZipper's SEO traffic. **Why this matters for GameZipper**: (a) the existing G-category tests verify that a game RENDERS and that the canvas is non-empty after interaction (px:true in the Kachilu eval pattern), but do NOT verify the responsiveness latency between interaction and visual response; (b) GameZipper's sync-game-loop pattern (per the Canvas game lazy-init pitfalls in R326) and the heavy WebGL rendering path mean individual game frames can take >200ms when synchronously processing complex physics or animation — this would yield poor INP without any visible bug. (c) Tests should include a deliberate click→response measurement: `performance.now()` before and after a click event, asserting the diff is <200ms in the fast-path and <500ms in the worst-case path. (d) The Event Timing API is supported in Chrome / Edge / Firefox / Opera but NOT Safari — for Safari users, INP is reported as N/A; this means GameZipper's INP-derived SEO scores are computed from a Chromium-only subset, but Safari users still experience the same latency. (e) for QA: include INP benchmarks as part of the per-game QA output, with the delta vs the 200ms "Good" threshold as a metric in the games-list.json so trends can be tracked across versions. (f) the modern 2026 frontend performance standard treats INP <200ms as the baseline for any interactive page — GameZipper games targeting this threshold will rank better and convert better in ad-impression context. (g) GameZipper games that use `requestAnimationFrame` patterns (the R333 / R335 lazy-init) may exceed 200ms INP on the first interaction because the rAF chain delays measurement — tests should specifically time the SECOND interaction (when rAF is already running) rather than the first.
