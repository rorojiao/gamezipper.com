- [P-165] **[P2]** WebGPU + WebAssembly (WASM) Synergy Pattern (2026 Industry Guidance) — WASM Game Modules Need Explicit Memory Pools and Pre-Allocation to Avoid GC-Induced Frame Stalls on Long Sessions; Pattern for Future GameZipper Games Built in Rust / C++ Compiled to WASM (tool.lu/en_US/article/7yi/detail Jun 11 2026 — "性能突破:WebGPU WebAssembly(WASM)技术深度解析...WebGPU与WebAssembly(WASM)协同发力,突破Web端性能瓶颈...WASM处理CPU密集型任务,WebGPU加速GPU计算,实现高效联动...优化策略包括共享内存､异步调度和资源预分配"). **Why this is the WASM-NEEDS-MEMORY-POOL dimension distinct from P-164 (CPU Performance API) and P-163 (V8/WebGPU overhaul)**: P-164 is device-class detection API; P-163 is engine overhaul. P-165 captures the WASM-MEMORY-PATTERN-GUIDANCE class where 2026 industry articles (this one from tool.lu) prescribe specific memory-management patterns for games that combine WebGPU rendering with WASM logic — the patterns include explicit memory pools, pre-allocation, async scheduling, and shared memory between WASM and WebGPU. The reason: WASM modules don't have JS-style GC, but they allocate from a linear memory that, when grown (via `memory.grow()`), can cause frame stalls; and JS-side references to WASM memory buffers can hold WASM memory alive past usefulness, bloating the heap. **Why this matters for GameZipper**: (a) GameZipper today is pure JS + Canvas/WebGL, so this is forward-looking — but the 2026 trend (per the article) is Rust/C++ game logic compiled to WASM for 2-5x CPU performance, which would matter for any compute-heavy game (physics simulation, large-scale particle systems, AI opponents); (b) if/when GameZipper adopts a WASM-based game (e.g., a Unity-HTML5 export, or a Rust+wgpu game), the naive memory pattern (allocate per frame) will cause periodic jank that pure-JS games don't exhibit — players will report "the game stutters every few seconds"; (c) the WebGPU side has its own memory model — buffer uploads must happen before render pass, not during (per playcanvas/engine PR #5716 from search results); (d) the article's prescribed patterns (memory pool, async scheduling, resource pre-allocation) are exactly the patterns GameZipper QA should check for when first encountering a WASM-based game submission. **QA methodology**: (1) when a WASM-based game appears in the pipeline, run a 5-minute sustained session and capture a Chrome DevTools Performance trace — look for periodic ~50-100ms long tasks at memory.grow boundaries; (2) check the WASM module's source (if available) for an explicit memory pool / arena allocator, not naive per-frame `malloc`; (3) check the JS↔WASM boundary for buffer-copy patterns — `new Uint8Array(wasmMemory.buffer, ptr, len)` creates a view, not a copy, and holding these views prevents WASM memory from shrinking; (4) verify WebGPU buffer uploads happen in a `device.queue.writeBuffer()` call before `beginRenderPass()`, not interleaved; (5) for any WASM game, add a long-session (30min+) stability test to the QA gate, since memory pattern bugs manifest over time, not in a 2-minute smoke test. **Source**: tool.lu/en_US/article/7yi/detail (Jun 11 2026, WebGPU+WASM synergy deep-dive); blog.csdn.net/gitblog_00372/article/details/158277179 (Mar 4 2026, WebGPU/WASM perf optimization guide); blog.csdn.net/weixin_36282234/article/details/158302311 (May 20 2026, RMBG-2.0 WASM memory pool pattern).
