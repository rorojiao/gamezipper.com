- [S-384] **[P1]** Safari 26 Tracking Parameter Stripping — gclid/fbclid Removed from URLs Breaking Analytics/Ad Attribution (cloud.tencent.com/developer/article/2637644 Jul 11 2026 — "Safari版本26已经发布...Added support for filtering tracking parameters in links in regular browsing mode (134226694)...第一个134226694就是新版Safari将会自动移除gclid、fbclid里所以提及"). **Why this is the BROWSER-LEVEL-URL-TRACKING-PARAMETER-STRIPPING dimension distinct from B-197 (Safari ITP third-party cookie blocking — cookie-layer) and S-365 (Chrome Privacy Sandbox Topics API — interest-based targeting)**: B-197 blocks third-party COOKIES (stored data layer); S-365 replaces tracking COOKIES with Topics API (alternative mechanism); S-384 captures the URL-MODIFICATION layer where Safari doesn't just block cookies or offer alternatives — it ACTIVELY MODIFIES the URL by stripping `gclid`, `fbclid`, `msclkid`, `mc_eid` and other known tracking parameters from the URL BEFORE the page's JavaScript runs — `window.location.search` will NOT contain these parameters, `URLSearchParams.get('gclid')` returns `null`, and any analytics/ads script that reads them at page load will see nothing. **Why this matters for GameZipper**: (a) if GameZipper uses Google Ads UTM/gclid parameters for campaign tracking, Safari 26 users arriving via ad clicks will have the gclid stripped — Google Ads conversion tracking will report ZERO conversions from Safari 26 users, making ad campaigns appear ineffective when they're actually working; (b) if any game uses URL parameters for level codes, share links, or referral IDs that happen to match a tracking parameter pattern, Safari will strip them — e.g., a share link like `gamezipper.com/game/?gclid=abc` will lose the parameter; (c) Monetag/AdSense attribution may be affected if they rely on URL-based click IDs for revenue attribution — ad revenue from Safari 26 traffic may be misattributed or lost; (d) this is REGULAR MODE, not Private/Incognito — ALL Safari 26 users are affected by default; (e) the stripping happens before `DOMContentLoaded`, so no JavaScript workaround can recover the original URL. **QA methodology**: (1) open GameZipper in Safari 26+; (2) navigate to a game page with a URL like `gamezipper.com/game/?gclid=test123&fbclid=test456`; (3) check `window.location.search` — if `gclid` and `fbclid` are MISSING, Safari stripped them; (4) check Google Analytics real-time — confirm the visit is recorded but without ad click attribution; (5) test with custom parameters (`?level=5&ref=friend`) to confirm only tracking params are stripped, not all params; (6) if GameZipper uses gclid for ad attribution, implement server-side attribution via Google Ads click redirect as a fallback (the redirect happens server-side before Safari can strip the parameter); (7) document this as a known limitation in analytics reporting.

P-057 is a WASM failure mode (memory exhaustion); W-325 captures the PLATFORM-CAPABILITY evolution where WASM 2.0's SharedArrayBuffer-based multithreading enables TRUE PARALLEL COMPUTATION in browser games — multiple WASM threads can share memory and compute physics/AI/pathfinding simultaneously, eliminating the single-threaded bottleneck that has limited ALL JS browser games to sequential execution on the main thread. **Why this matters for GameZipper**: (a) GameZipper's physics-heavy games (water sort solver, block stacking, ragdoll) currently run ALL computation on the main thread, causing frame drops during complex calculations — WASM 2.0 threads enable offloading physics to worker threads with ZERO main-thread impact; (2) for physics-heavy games, profile main-thread time with Chrome DevTools Performance tab — if >50% of frame budget is spent on JS computation, it's a candidate for WASM thread migration; (3) verify SharedArrayBuffer availability: `typeof SharedArrayBuffer !== 'undefined'` — returns `true` only if cross-origin isolation is active; (4) benchmark a WASM-compiled physics module vs the JS equivalent on GameZipper's target devices (low-end Android); (5) check if cross-origin isolation breaks any third-party scripts (Monetag, AdSense) — some ad networks don't send `Cross-Origin-Resource-Policy` headers and will be blocked under COEP.
