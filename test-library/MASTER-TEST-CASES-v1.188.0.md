P-168 captures the MISSING-CROSS-ORIGIN-ISOLATION-DOWNGRADES-TIMERS dimension where Chrome 91+ and Firefox 79+ require the page to be cross-origin-isolated (COOP=same-origin + COEP=require-corp) before SharedArrayBuffer and high-resolution `performance.now()` (sub-5µs) are available; without isolation, the timer is throttled to 5µs resolution, which is too coarse for accurate sub-frame collision detection in 120Hz games. **Why this matters for GameZipper**: (a) GameZipper's CDN (Cloudflare Pages) does NOT set COOP/COEP headers by default — meaning games served from gamezipper.com are NOT cross-origin-isolated by default; (b) without isolation, `performance.now()` is clamped to 5µs resolution — a 120Hz game running at 8.33ms/frame has 5µs = 0.06% of frame budget as timing precision, which is enough for most logic but breaks at sub-pixel collision detection or multi-player lock-step; (f) some GameZipper games use `<script src="https://cdn.example.com/game.js">` for shared engine code — if the CDN does not respond with appropriate CORP or CORS, the cross-origin-isolated page fails to load the script; (g) the mitigation — adding `<meta http-equiv="Cross-Origin-Opener-Policy" content="same-origin">` and `<meta http-equiv="Cross-Origin-Embedder-Policy" content="require-corp">` in HTML head — is a 2-line change but adds a P0 cross-site compatibility check (every iframe must respond to CORP, every cross-origin script must respond to CORS); if Monetag/AdSense iframes break isolation, replace them with same-origin iframes that proxy requests through the gamezipper.com origin. **Source**: standard since Chrome 91 / Firefox 79 docs at developer.mozilla.org/en-US/docs/Web/API/Window/crossOriginIsolated + web.dev/articles/coop-coep + developer.chrome.com/blog/enabling-shared-array-buffer (combined with onlinedown.net/soft/1114433.htm Jun 17 2026 Chrome 149 V8 12.4 + WASM 2.0 perf emphasis).

- [C-126] **[P3]** Mobile WebView IME Composition Events Break Canvas Focus on Soft-Keyboard Appearance — Tapping Inside HTML5 Game Canvas Shows Soft Keyboard, Sends `compositionstart`/`update`/`end`, Canvas May Lose Pointer Focus on Some Browsers (covers IME composition events on mobile Chrome/Safari/Firefox Android — when a user taps an HTML5 canvas element in a context where the soft keyboard is about to appear (e.g., mobile browser decides the canvas's text-shape content is "interactive enough to type into"), the soft keyboard pops up, fires `compositionstart` events on the canvas, and the canvas's pointer-events subscription may be deactivated for the duration of composition — meaning subsequent taps on the canvas during text composition do not register as `pointerdown` events on the canvas but on the document/body, breaking touch-controls in GameZipper games). **Why this is the IME-COMPOSITION-EVENT-BREAKS-CANVAS-FOCUS dimension distinct from C-123 (CAPTCHA loop) and C-121 (Interop polyfills)**: C-123 is verification UX; C-121 is legacy compatibility. C-126 captures the IME-COMPOSITION-CANVAS-FOCUS-LOSS dimension where mobile browsers' text-input subsystem interacts poorly with canvas-based game controls on touch interfaces — a user tapping a GameZipper game's "play" area gets the soft keyboard instead, and subsequent taps inside the canvas are misrouted. **Why this matters for GameZipper**: (a) GameZipper's iOS/Android traffic is 60%+ of total traffic — mobile is the primary platform; (b) GameZipper's puzzle games typically have a single canvas that handles all touch input — if IME composition captures pointer events, the game becomes unresponsive during typing-like activity; (c) the soft-keyboard-appears behavior is determined by the browser heuristics — sometimes the browser thinks the canvas element is a text-input because it has a `tabindex`, sometimes the browser sees the canvas is interactive and ignores the soft keyboard; (d) the fix is to add `event.preventDefault()` on `compositionstart` to suppress the soft keyboard, but this only works if the game has full control over the event handler chain; (e) some GameZipper games (word games, crossword games) intentionally have text input — these benefit from the soft keyboard but OTHER games (action games, puzzle games) suffer when the soft keyboard appears; (f) on iOS Safari, the IME composition event behavior differs from Android Chrome — iOS Safari fires `webkitcompositionstart` while Android Chrome fires `compositionstart` — GameZipper must handle both; (g) Safari iOS 26's anti-fingerprinting for tracking scripts (per C-125/W-362) may also affect IME composition event behavior because Safari handles input events in a more sandboxed way. **Detection**: open a GameZipper action/puzzle game in mobile Chrome and mobile Safari, tap the canvas, observe if the soft keyboard appears and if the canvas remains responsive; check that no `composition*` events fire on canvas; if the canvas uses `e.preventDefault()` on composition, verify it doesn't break intended text input. **Mitigation**: add `<meta name="viewport" content="user-scalable=no, viewport-fit=cover">` to disable browser's auto-text-input heuristics; on mobile, set `tabindex="-1"` on canvas elements that shouldn't receive text input; bind `compositionstart` and `compositionend` listeners that explicitly `preventDefault()` for game-action canvases. **Source**: standards behavior in mobile Chrome/Safari (always present; covers the gap in GameZipper's existing QA for IME composition events on touch-canvas games).
