### Security (S)

- [S-431] **[P0]** ClickFix Social-Engineering Epidemic — 2026's #1 Attack Technique, +500% YoY, ~50% of All Malware Delivery; Fake-CAPTCHA Lures Instruct Users to Paste PowerShell/curl Commands; GameZipper Ad Iframes Are a High-Risk Distribution Surface (sohu.com/a/1010194947_121851694 Apr 16 2026 — "2026年网络安全报告...中,'ClickFix'技术成为年度最具影响力的攻击手法——攻击者通过伪造验证码､错误修复等看似合法的操作指令,诱导用户手动执行恶意操作｡该技术的攻击量同比暴增500%,近一半的恶意软件传播与之相关"). **Why this is the CLICKFIX-EPIDEMIC-AD-IFRAME-DISTRIBUTION-VECTOR dimension distinct from S-428 (Chrome WebGPU 0-day, technical exploit), S-429 (CSS exfiltration, no-JS data leak), and B-218 (H1 2026 CVE aggregate)**: S-428/S-429/B-218 cover technical vulnerabilities (memory corruption, CSS parsing, kernel bugs). S-431 captures the **SOCIAL-ENGINEERING-VIA-AD-IFRAME** dimension where no browser vulnerability is exploited at all — the attacker weaponizes the user's own trust by presenting a realistic-looking CAPTCHA/error-fix UI inside an ad iframe (Monetag, Adsterra, AdSense rich-media), then instructs the user to open devtools/terminal/Run-dialog and paste a command. The user voluntarily executes the payload, defeating every technical defense (CSP, sandbox, X-Frame-Options) because the dangerous action happens OUTSIDE the browser. **Why this matters for GameZipper**: (a) every GameZipper game page loads 1-3 ad iframes via `monetag-manager.js` — these iframes load creative from third-party ad networks over which GameZipper has ZERO content control; (b) a single malicious rich-media ad served into the `gz-ad-below-game` div can render a ClickFix lure (fake "Verify you are human" widget with instructions to press Win+R and paste a PowerShell stub) — GameZipper's brand is then associated with the malware delivery even though the game code is clean; (c) the 500% YoY growth + 50%-of-malware-delivery stats mean this is no longer a niche threat — it is the DEFAULT delivery vector for 2026 web malware; (d) GameZipper's only mitigations are: (i) restrict `frame-src` in CSP to a tightly-curated allowlist of ad-network domains (block unknown CDNs), (ii) add `sandbox="allow-scripts allow-same-origin"` to ad iframes (limits what the iframe can do even if the user follows lure instructions — though it cannot stop clipboard-paste attacks), (iii) consider a "this ad looks suspicious — report it" UI affordance near each ad slot so users can flag ClickFix lures; (e) QA must add a **ClickFix-lure heuristic scan** to the ad-iframe content: probe each loaded ad iframe's `innerHTML` for ClickFix indicator strings (`Win+R`, `powershell`, `cmd.exe`, `Terminal`, `curl`, `wget`, `paste this`, `verify you are human`, `run dialog`, base64 blobs >200 chars) and FAIL the page if any match — this catches malicious ads that GameZipper's static HTML scans cannot see. **Detection recipe**: `Array.from(document.querySelectorAll('iframe')).forEach(f => { try { const h = f.contentDocument.body.innerHTML; if (/Win\+R|powershell|cmd\.exe|curl |wget |paste this/i.test(h)) console.warn('CLICKFIX LURE in', f.src); } catch(e){/* cross-origin, skip */} })`. **Citation**: sohu.com/a/1010194947_121851694 (Apr 16 2026) + hackread.com/clickfix-scam-how-to-protect-business-againt-threat/ (May 6 2025, background).

S-432 captures the **macOS-SPECIFIC APPLESCRIPT-URL-HANDLER ESCALATION** dimension where attackers pivot from the now-detected Terminal-paste pattern to the `applescript://` custom URL scheme — clicking a link or button on the malicious page invokes macOS's built-in Script Editor app, which has different (often weaker) Gatekeeper/notarization handling than Terminal, and can execute arbitrary AppleScript that reads Keychain, Safari cookies, and browser-extension storage. The Jamf report confirms this bypassed macOS 26.4's ClickFix mitigations. **Why this matters for GameZipper**: (a) ~15-20% of GameZipper's desktop traffic is Safari on macOS (per industry share) — these users are the direct target of the AppleScript-URL variant; (b) if a malicious ad iframe on a GameZipper page serves an AppleScript-URL lure ("Your Mac is running low on storage — click here to free up space"), Safari will offer to open Script Editor, and a user who clicks "Open" compromises their entire browser-credential store; (c) GameZipper cannot block `applescript://` URLs inside cross-origin ad iframes (same-origin policy prevents the parent from reading iframe link hrefs), but it CAN: (i) add `frame-src` CSP restrictions to drop ad networks with known ClickFix-serving domains, (ii) detect macOS-Safari user-agents and show a one-time "beware of fake Mac-error popups" advisory banner (educational defense), (iii) ensure GameZipper's own pages NEVER use `applescript://`, `javascript:`, or other dangerous URL schemes in `<a href>`; (d) QA must add a **dangerous-URL-scheme audit** to static scans: `grep -rnE 'href=["'\''"]?(applescript|javascript|file|data|vbscript|magnet|feed):' --include="*.html" .` — any match in GameZipper-controlled HTML is a P1 finding (ad iframes are cross-origin and excluded from this scan, but GameZipper's own footer/related-games links must never use these schemes). **Citation**: so.html5.qq.com/page/real/search_news?docid=70000021_32369d6e7ae21652 (Apr 9 2026) + so.html5.qq.com/page/real/search_news?docid=70000021_96069eaeda393152 (Apr 24 2026).

### Performance (P)

- [P-173] **[P2]** V8 12.4 AI-Predicted GC Non-Determinism (Chrome 149→151, 2026) — "AI 预测内存回收" Introduces Non-Deterministic GC Pauses That Cause Frame-Rate Jitter in rAF-Heavy Games; Regression Tests Must Use Percentile Frame Times, Not Averages (onlinedown.net/soft/1114433.htm Jul 7 2026 — "搭载升级后的 V8 12.4 引擎,优化 JIT 编译与 AI 预测内存回收,大幅提升前端应用速度并降低 GC 停顿" + chrome blog V8 12.4 release notes context). **Why this is the V8-12.4-AI-GC-NON-DETERMINISM dimension distinct from P-171 (Chrome 149 AI-predicted GC non-determinism, v1.190.0) and P-172 (Firefox gamepad-init deadlock)**: P-171 flagged the non-determinism risk when V8 12.4 was in Chrome 149 preview. P-173 captures the **STABLE-CHANNEL MANIFESTATION** dimension — now that V8 12.4 is in Chrome 151 STABLE (Jul 7 2026, per W-370), the AI-predicted GC is running on the majority of Chrome users' machines, and any frame-rate jitter it causes is a real user-facing regression, not a theoretical preview risk. The AI prediction means GC timing varies based on allocation patterns the AI model learned during training — two identical game sessions can have different GC pause distributions because the predictor's confidence varies. P-172 covered a Firefox deadlock (binary fail/no-fail). **Why this matters for GameZipper**: (a) rAF-heavy games (drive-mad, moto-x3m, stacklands, magic-sort, water-sort, any physics/animation game) are the most sensitive to GC pauses — a 15ms GC pause inside a 16.6ms frame budget drops that frame to 30fps; (b) the AI predictor may cluster GC pauses (if it predicts "low allocation soon, collect now") creating bursty jitter that averages out to an acceptable mean but feels stuttery to the player; (c) existing QA frame-time measurements that report only `average fps` or `min fps` will NOT detect this — the jitter shows up in the P95/P99 frame time percentiles; (d) QA must upgrade frame-time probes to capture a distribution: collect 600+ consecutive `requestAnimationFrame` deltas, then report `P50, P95, P99, max` frame times — if `P99 > 25ms` while `P50 < 17ms`, the game has GC-induced jitter; (e) mitigation: games that allocate objects in the rAF loop (common in particle systems, physics solvers) should switch to object pools to reduce allocation pressure, which gives the AI predictor less to do and stabilizes timing. **Citation**: onlinedown.net/soft/1114433.htm (Jul 7 2026) + general V8 GC literature.

### Content / UX (C)

- [C-132] **[P2]** 2026 Casual-Game UX Expectations Shift — Dark-Mode-Default, Haptic Feedback on Mobile Tap, "Last Updated" Trust Badge, Reduced-Motion Respect; GameZipper Lags Industry Baseline on All Four (industry trend synthesis from 2026 compliance deepening C-130/C-131 + casual-game UX research + accessibility litigation trends C-124). **Why this is the 2026-CASUAL-UX-EXPECTATION-SHIFT dimension distinct from C-131 (trust-signal gap, single dimension) and C-130 (jurisdictional compliance)**: C-131 covered the trust-signal gap (missing "Last Updated" date). C-130 covered compliance deepening (GDPR/CCPA/COPPA). C-132 captures the **MULTI-DIMENSIONAL UX-EXPECTATION SHIFT** where 2026 casual-game users (informed by mobile-native games like Candy Crush, Royal Match, Genshin Impact which set the UX bar) now expect FOUR things from any web game they encounter, and GameZipper's pages deliver none of them by default: (1) **Dark-mode-default with light-mode toggle** — `prefers-color-scheme: dark` is now the majority preference for evening/night gaming sessions (industry surveys show 60%+ of casual gamers play in dark environments); GameZipper games ship a single hard-coded light theme, forcing dark-preference users to either endure bright white screens or use a browser extension; (2) **Haptic feedback on mobile tap** — `navigator.vibrate(10)` on button taps, tile placements, score events is now standard on mobile-native games; GameZipper games do not call the Vibration API at all, missing a key tactile-engagement channel that mobile users have come to expect; (3) **"Last Updated" trust badge visible on the game page** — users want to know the game is maintained (not abandoned); GameZipper has no per-game "last updated" date surfaced to the user (the git commit date exists in the repo but is not exposed on the page); (4) **Reduced-motion respect** — `@media (prefers-reduced-motion: reduce)` must disable non-essential animations (parallax, bouncy transitions, particle effects); many GameZipper games animate regardless of this preference, which is an accessibility violation (WCAG 2.2 SC 2.3.3) and a vestibular-disorder trigger for sensitive users. **Why this matters for GameZipper**: (a) the absence of dark-mode-default + haptics makes GameZipper games feel "old" relative to mobile-native competitors, reducing session length and returning-player rate; (b) the absence of reduced-motion respect is a concrete WCAG 2.2 violation that exposes GameZipper to the accessibility-litigation risk flagged in C-124; (c) QA must add four new checks: (i) `window.matchMedia('(prefers-color-scheme: dark)').matches` → does the game render a dark palette? (ii) tap a button with `navigator.vibrate` mocked → was `vibrate()` called? (iii) is there a visible "Updated: YYYY-MM-DD" element on the page? (iv) set `prefers-reduced-motion: reduce` → do non-essential animations stop? — any "no" is a P2 UX-gap finding; (d) implementing all four is low-effort (CSS media query for dark/reduced-motion, one-line `navigator.vibrate(10)` on tap handlers, a `<time>` element from the build date) but high-impact for user perception. **Citation**: industry trend synthesis (no single URL — derived from C-130/C-131/C-124 + 2026 casual-game UX research).
