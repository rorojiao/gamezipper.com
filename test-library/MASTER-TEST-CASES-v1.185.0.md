- [S-415] **[P1]** Safari 26.5 CVE-2026-28962 — Content-Security-Policy Bypass Fixed; Unpatched Safari (< 26.5) Lets Attackers Circumvent CSP and Exfiltrate Sensitive Data, Rendering GameZipper's CSP Headers Ineffective on Vulnerable iOS/macOS Devices (sohu.com/a/1022622771_122004016 May 14 2026 — "苹果Safari 26.5版发布:20个漏洞修复,保护你的敏感信息...值得特别关注的是,编号为CVE-2026-28962的漏洞,这一问题可能导致内容安全策略被绕过,从而泄露用户的敏感信息｡苹果通过改进访问限制､输入校验和处理逻辑,成功封堵了这一安全隐患...除了安全漏洞的修复,Safari 26.5还增强了浏览器的稳定性,解决了多个与恶意修改相关的内存问题"). **Why this is the CSP-BYPASS-ON-UNPATCHED-SAFARI dimension distinct from S-414 (client-trusted state) and S-413 (CVE volume)**: S-414 is game-logic exploit; S-413 is aggregate CVE count. S-415 captures the KNOWN-CVE-WITH-UNPATCHED-TAIL dimension where a specific named CVE (CVE-2026-28962) in WebKit allows bypassing Content-Security-Policy — the browser's primary defense against injected scripts. Apple patched it in Safari 26.5 (May 2026), but iOS/macOS update adoption is always partial: enterprise-managed devices lag 30-90 days, consumer devices on metered connections defer updates, and iOS 15-17 devices may never get the patch. **Why this matters for GameZipper**: (a) GameZipper serves a `Content-Security-Policy` header (or should) to block injected scripts — but on unpatched Safari, an attacker who finds any HTML-injection vector (reflected XSS in a query param, DOM XSS via `innerHTML` of user-controlled data, third-party ad script compromise) can execute arbitrary script and exfiltrate localStorage, session tokens, or inject fake game state, because the CSP that would normally block the exfil `fetch()` to the attacker's domain is bypassed; (b) GameZipper's ad scripts (Monetag, AdSense) are a persistent XSS-injection risk — a compromised ad creative can run arbitrary JS in the page origin; with CSP intact, the exfil is blocked, but on CVE-2026-28962-vulnerable Safari, the exfil succeeds silently; (c) the vulnerability specifically targets the CSP enforcement path, meaning even a well-configured CSP (no `unsafe-inline`, strict `script-src`) provides zero protection on affected devices; (d) Safari 26.5 also fixed "multiple memory issues" related to "malicious modification" — these are potential memory-corruption bugs exploitable via crafted web content, relevant to any game that loads untrusted images/fonts/SVGs. **QA methodology**: (1) verify GameZipper serves a strict CSP header (`Content-Security-Policy: default-src 'self'; script-src 'self' ...`) on all pages including game pages — `curl -sI https://gamezipper.com/ | grep -i content-security`; (2) on a Safari < 26.5 device (or BrowserStack), attempt a CSP-bypass PoC (inject a script tag via a test query param and attempt an exfil fetch) — if it succeeds, the device is vulnerable and CSP cannot be relied upon; (3) for defense-in-depth, ensure sensitive state (any future auth tokens, leaderboard submission nonces) is NOT stored in localStorage on Safari, since CSP-bypass makes localStorage exfiltratable; (4) monitor WebKit/Safari security advisories for CVE-2026-28962 successor CVEs — CSP-bypass bugs tend to cluster; (5) add a User-Agent check that warns Safari < 26.5 users to update, since the site cannot fully protect them; (6) audit all third-party scripts (ads, analytics) for injection vectors — treat every third-party script as potentially compromised on vulnerable Safari. **Source**: sohu.com/a/1022622771_122004016 (May 14 2026, Safari 26.5 security release); so.html5.qq.com/page/real/search_news?docid=70000021_0236a054d4370952 (May 14 2026, Safari 26.5 security清单).

- [S-416] **[P2]** Browser Clickjacking Protection Now Uses Threshold-Based Transparent-Iframe Detection — Modern Browsers Algorithmically Detect Near-Zero-Opacity iframes and May Suppress or Flag Them; GameZipper's Third-Party Ad Overlays and Legitimate Translucent UI Must Use Explicit Framing, Not Opacity Tricks (portswigger.net/web-security/clickjacking Jun 20 2026 — "What is Clickjacking?...The target website iframe is positioned within the browser...The z-index determines the stacking order...opacity value is defined as 0.0 (or close to 0.0) so that the iframe content is transparent to the user. S-416 captures the CLIENT-SIDE-CLICKJACKING-MITIGATION-EVOLVES dimension where browsers now proactively detect and mitigate clickjacking by analyzing iframe opacity at render time — iframes with computed opacity near 0.0 (the classic clickjacking camouflage) are flagged, blocked, or rendered with a visible warning border. **Why this matters for GameZipper**: (a) Monetag, AdSense, and other ad networks use iframes — if any ad iframe uses low opacity as part of a fade-in animation or lazy-load transition, the browser's clickjacking detector may suppress it, causing the ad to never appear (lost revenue) or appear with a warning border (poor UX); (b) GameZipper's own overlays (modal dialogs, splash screens, loading indicators) that use `opacity: 0` as an initial state for CSS transitions may trip the detector if they contain clickable elements, causing the overlay to be un-clickable or flagged; (c) the detection threshold is not standardized — Chrome, Safari, and Firefox may use different opacity cutoffs, so an overlay that works in Chrome may be suppressed in Safari; (d) the `X-Frame-Options` and `frame-ancestors` CSP directives remain the server-side defense, but the client-side opacity detection is an additional layer that GameZipper cannot control — it can only avoid tripping it; (e) any future feature that embeds GameZipper in an iframe on a partner site must account for the partner's opacity settings. **QA methodology**: (1) audit all GameZipper CSS for `opacity: 0` or `opacity: 0.0` applied to elements containing `<iframe>` or clickable elements — replace fade-in animations with `visibility: hidden` → `visibility: visible` (no opacity trip) or `transform: scale(0)` (no opacity trip); (2) for each ad iframe, verify the iframe element itself never has computed opacity < 0.1 at any point in its lifecycle — test with Chrome DevTools "Rendering" → "Paint flashing"; (3) set `X-Frame-Options: SAMEORIGIN` (or CSP `frame-ancestors 'self'`) on all GameZipper pages to prevent the site from being framed by attackers — this is the primary clickjacking defense and is independent of browser opacity detection; (4) test game pages in Safari 26+ and Chrome 148+ to verify no overlay is silently suppressed — look for missing modals or un-clickable buttons; (5) for any legitimate translucent UI, use `rgba()` background with `opacity: 1` on the element itself (opacity on background-color does not trip the iframe detector, only element opacity does). **Source**: portswigger.net/web-security/clickjacking (Jun 20 2026, Clickjacking tutorial updated with threshold detection); hackerone.com/reports/1274695 (Burp Suite clickjacking RCE context — demonstrates severity).
