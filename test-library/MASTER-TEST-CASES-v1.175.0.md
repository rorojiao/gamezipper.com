- [W-331] **[P2]** Safari 26.4 CSS Grid Lanes (Native Masonry/Waterfall) — Browser-Native Layout Conflicts with JS Masonry Libraries (so.html5.qq.com/page/real/search_news?docid=70000021_17369c3312871752 Mar 25 2026 — "苹果更新Safari 26.4浏览器:回归紧凑标签布局、修复 191 项 BUG...在新功能方面,重点引入了 CSS Grid Lanes(CSS 网格轨道)技术｡该技术为开发者构建复杂的视觉画廊和高级网页排版,提供了一个完美解决方案" + new.qq.com/rain/a/20251228A034KJ00 Dec 28 2025 — "WebKit 浏览器引擎上周(12月19日)更新开发文档,提出 CSS Grid Lanes 概念,旨在将瀑布流(Waterfall)排版放入浏览器原生 CSS 中｡官方文档中显示,CSS Grid Lanes 方案基于 Mozilla 与 WebKit 的早期工作"). **Why this is the NATIVE-MASONRY-LAYOUT dimension distinct from W-330 (HTML-in-Canvas DOM composited into canvas) and W-329 (Edge Copilot UI convergence)**: W-330 is about DOM rendered INTO canvas; W-329 is about browser chrome styling; W-331 captures the NATIVE-LAYOUT-ENGINE-MASONRY class where Safari 26.4 now natively implements CSS Grid Lanes (a.k.a. CSS masonry / waterfall layout) — previously, masonry/waterfall layouts required JavaScript libraries (Masonry.js, Isotope, Packery) or CSS `columns` hacks. **Why this matters for GameZipper**: (a) the GameZipper homepage game grid uses a custom responsive grid — if it ever adopts CSS Grid Lanes (or if a theme/CSS framework auto-applies `grid-template-rows: masonry`), Safari 26.4 will lay it out natively while Chrome/Firefox ignore the rule, causing visual divergence; (b) game "level select" screens with variable-height cards (different cover image aspect ratios) may look correct in Safari 26.4 masonry but broken (large gaps) in non-supporting browsers if a developer assumes masonry works everywhere; (c) JS masonry libraries running on top of Safari 26.4's native masonry will DOUBLE-APPLY layout (library measures DOM → reflows → native engine also reflows → infinite reflow loops or mispositioned elements); (d) `grid-auto-flow: masonry` and related properties are not feature-detectable via `@supports` in older Safari, so progressive enhancement patterns fail silently; (e) the CSS Working Group is still finalizing the masonry syntax — Safari's implementation may change, breaking layouts that shipped against the current syntax. **QA methodology**: (1) feature-detect via `CSS.supports('grid-template-rows', 'masonry')` — verify games handle both true/false correctly; (2) open the GameZipper homepage and any level-select gallery in Safari 26.4 vs Chrome vs Firefox — screenshot-compare for layout drift; (3) if a JS masonry library is loaded, verify it doesn't run when native masonry is active (guard with feature detection); (4) test `ResizeObserver` callbacks on masonry-laid-out elements — Safari may fire resize events at different times than the JS library expects; (5) check that card click/touch targets remain accurate after native masonry repositions elements (hit-testing must use live `getBoundingClientRect()`, not cached positions).

- [W-332] **[P2]** Safari 26.4 WebTransport Support — New Transport API Feature Detection & Fallback Verification (so.html5.qq.com/page/real/search_news?docid=70000021_17369c3312871752 Mar 25 2026 — "苹果更新Safari 26.4浏览器...同时,新版本正式支持了 WebTransport 技术"). **Why this is the NEW-NETWORK-TRANSPORT-API dimension distinct from W-331 (CSS Grid Lanes — layout) and B-197 (Safari ITP cookie blocking)**: W-331 is layout; B-197 is cookies; W-332 captures the QUIC-BASED-BIDIRECTIONAL-TRANSPORT class where Safari 26.4 adds WebTransport — a modern alternative to WebSocket built on HTTP/3 (QUIC), supporting both reliable streams and unreliable datagrams. **Why this matters for GameZipper**: (a) any GameZipper multiplayer or real-time game feature that feature-detects `typeof WebTransport !== 'undefined'` to choose transport will now see WebTransport available in Safari 26.4 — but Safari's WebTransport implementation may have different conformance than Chrome's (different datagram size limits, different error codes, different certificate verification for serverCertificateHashes); (b) Safari's WebTransport may not support the same set of options (e.g., `allowPooling`, `congestionControl` hints) that Chrome supports — code assuming Chrome's option set will fail on Safari; (c) WebTransport over HTTP/3 requires the server to support QUIC — if GameZipper's CDN (Cloudflare) terminates HTTP/3 but not WebTransport, the connection fails with a non-obvious error; (d) unreliable datagrams (used for game state updates where dropped packets are OK) may behave differently — Safari may drop more aggressively or buffer differently than Chrome; (e) the fallback to WebSocket must be tested — a game that uses WebTransport when available and WebSocket otherwise must verify the WebSocket path still works on Safari 26.4 (don't assume WebTransport is always the right choice). **QA methodology**: (1) feature-detect `window.WebTransport` in all browsers — map the support matrix; (2) if any game uses WebTransport, test in Safari 26.4 specifically — connect, send datagrams, receive datagrams, close; (3) compare option support: `serverCertificateHashes`, `allowPooling`, `congestionControl` — Safari may reject unsupported options; (4) test the fallback: disable WebTransport (feature-detect to false) and verify WebSocket path still functions; (5) check error handling — Safari may emit different error events (`WebTransportError` vs generic) on connection failure.

### Security (S)

- [S-399] **[P1]** npm v12 Default-Disable Install Scripts — Build Pipeline Breaks on Dependencies Requiring postinstall (cloud.tencent.com/developer/article/2699338 Jul 2026 — "前端周刊2026W26...Node.js方面v26.4和v26.3双版本更新,npm v12默认禁止安装脚本引发安全讨论"). **Why this is the BUILD-PIPELINE-INSTALL-SCRIPT-BLOCK dimension distinct from S-400 (npm supply chain worm — runtime compromise) and S-397 (Chrome one-time media permissions)**: S-400 is runtime package compromise; S-397 is browser permission; S-399 captures the BUILD-TIME-SECURITY-DEFAULT class where npm v12, as a security hardening measure, defaults to NOT executing package install scripts (postinstall, preinstall, install) unless explicitly allowed via `--ignore-scripts=false` or a trusted-packages allowlist. **Why this matters for GameZipper**: (a) any dev/build dependency that relies on a `postinstall` script to compile native addons (e.g., `node-canvas`, `sharp`, `bcrypt`, `sqlite3`) or download prebuilt binaries (e.g., `puppeteer` downloading Chromium, `playwright` downloading browser binaries) will SILENTLY install without those steps completing — the package appears installed but its binary/native code is missing; (b) the GameZipper build pipeline (if it uses any node-based tooling — image optimization, sprite packing, OG image generation) may break on `npm ci` in a fresh CI environment with npm v12, with non-obvious errors like "module not found" or "native module compile failed" that appear to be code bugs but are actually missing install-script output; (c) developers running `npm install` locally may see different behavior than CI (local npm may be v10/v11 with scripts enabled, CI may be v12 with scripts disabled); (d) the `--ignore-scripts` default was a security decision — re-enabling globally weakens the security posture, so the correct fix is per-package allowlisting, not blanket re-enable; (e) transitive dependencies may also need install scripts — allowlisting only direct deps misses the transitive case. **QA methodology**: (1) audit `package.json` + `package-lock.json` for dependencies with install scripts: `npm query --json | jq 'to_entries[] | select(.value.scripts | keys | any(. | test("^(pre)?install$|postinstall"))) | .key'`; (2) run `npm ci` in a clean environment with npm v12 — verify the build still completes; (3) for each dependency that needs install scripts, add to `npm` config: `npm config set allow-scripts=true` (scoped) or use the `--allow-script` flag per package; (4) verify that the GameZipper static site build (which is vanilla JS, no build step) is unaffected — confirm no node tooling is in the critical path; (5) document the allowlist in the repo so CI and local dev match.

- [S-400] **[P1]** npm Supply Chain — Shai-Hulud Self-Replicating Worm + TanStack Targeted Attack (reversinglabs.com/blog/shai-hulud-worm-npm Sep 15 — "ReversingLabs (RL) researchers detected a first of its kind self-replicating worm on the npm open-source registry...The worm, named 'Shai-hulud,' spreads through a cascading compromise of npm accounts that inserts the malicious worm code into legitimate public and private npm packages belonging to the compromised developer...hundreds of npm packages that have been compromised...including ngx-bootstrap (300k weekly downloads), ng2-file-upload (100k weekly downloads) and @ctrl/tinycolor (2.2M weekly downloads)" + cloud.tencent.com/developer/article/2679559 Jul 2026 — "TanStack npm供应链攻击事件"). **Why this is the NPM-SUPPLY-CHAIN-WORM-AND-TARGETED-ATTACK dimension distinct from S-399 (npm install scripts blocked — build-time) and S-392 (Gemini extension privilege escalation)**: S-399 is build-time script blocking; S-392 is browser extension; S-400 captures the RUNTIME-DEPENDENCY-COMPROMISE class where two distinct attack patterns — (1) a self-propagating worm that compromises npm accounts and injects malicious code into legitimate packages, and (2) a targeted attack on the TanStack ecosystem — demonstrate that ANY npm dependency can become malicious at any time, even from reputable maintainers. **Why this matters for GameZipper**: (a) GameZipper is a static site with minimal npm dependencies, BUT any dev/build tooling (linters, formatters, image optimizers) in the repo is a vector — a compromised tool can inject malicious code into the built output (e.g., a compromised minifier could inject a skimmer into the production JS); (b) the Shai-Hulud worm spreads by compromising maintainer npm accounts — even "trusted" packages with millions of weekly downloads were compromised; (c) `@ctrl/tinycolor` (2.2M weekly downloads) is a color manipulation library that could easily be a dependency of a game framework or UI library GameZipper uses — a malicious version could exfiltrate data or hijack the canvas; (d) the TanStack attack (targeting React Query, Table, etc.) shows that even well-maintained ecosystems are targeted — if GameZipper ever adopts a React/TanStack-based admin or tooling, the risk applies; (e) npm's lack of mandatory 2FA for maintainers and the ease of typosquatting make this an ongoing, not one-time, risk. **QA methodology**: (1) run `npm audit --audit-level=moderate` regularly and review ALL findings; (2) pin ALL dependencies to exact versions (not `^` or `~`) in `package.json` — use `npm shrinkwrap.json` or `package-lock.json` with `npm ci` to prevent surprise updates; (3) use `npm ls --all` to inspect the full dependency tree for unexpected packages; (4) monitor for new npm advisories via `npm audit signatures` (npm's package signing) and tools like `socket.dev` or `snyk`; (5) verify the production static output (the actual HTML/JS served to users) does NOT include any node_modules or build-time-only code — GameZipper serves plain static files, so runtime npm risk is near-zero, but build-time injection risk remains; (6) maintain a SBOM (software bill of materials) for any build tooling.

- [P-154] **[P2]** Node.js 26 Temporal API Enabled by Default — Modern Date/Time API Conflicts with Polyfills & Browser Code (donews.com/news/detail/8/6543521.html May 5 2026 — "Node.js团队正式发布Node.js 26.0.0当前版本...新版默认启用现代Temporal日期/时间API,升级V8引擎至14.6.202.33,并集成Undici 8.0.2 HTTP客户端"). **Why this is the TEMPORAL-API-DEFAULT-ENABLED dimension distinct from P-152 (Chrome V8 engine overhaul — speed) and P-153 (Battery-saver tab freeze)**: P-152 is speed; P-153 is tab freezing; P-154 captures the NEW-DATE-TIME-API-STANDARDIZED class where Node.js 26 enables the Temporal API (the modern replacement for the `Date` object, providing immutable, timezone-aware, ISO-8601-compliant date/time types) by default — this is a server-side standard, but it affects game development because polyfills and browser code may conflict with it. **Why this matters for GameZipper**: (a) any server-side game logic (leaderboard timestamping, daily challenge scheduling, cooldown tracking) that uses Temporal on Node 26 will NOT work in browsers unless a Temporal polyfill is loaded — the temporal-polyfill or @js-temporal/polyfill must be shipped to the client, increasing bundle size; (b) code that feature-detects `typeof Temporal !== 'undefined'` to decide whether to load a polyfill must be careful — Node 26 reports Temporal as defined, but the browser may not, so the same code path takes different branches on server vs client; (c) existing `Date`-based code (countdown timers, "time since last save" displays) may have subtle bugs when migrated to Temporal — Temporal.Instant is in UTC by default while Date methods mix local and UTC; (d) the `Intl.DateTimeFormat` integration with Temporal differs from Date — game UIs showing localized dates/times may render differently; (e) Undici 8.0.2 (the new HTTP client) changes `fetch` behavior on the server — if GameZipper has any server-side rendering or API endpoints, the fetch semantics (headers, streaming, error handling) may differ from browser fetch. **QA methodology**: (1) audit all date/time usage: `grep -rn 'new Date\|Date\.now\|Temporal\.' --include="*.js"`; (2) if Temporal is used server-side, ensure a polyfill is shipped to the browser or that the code is server-only; (3) test countdown timers and cooldown logic across DST boundaries and timezone changes — Temporal handles these more correctly than Date, but only if the code uses Temporal's timezone-aware types (Temporal.ZonedDateTime, not Temporal.PlainDateTime); (4) verify that game replay/save timestamps stored as ISO strings round-trip correctly through both Temporal and Date parsing; (5) if using Undici's fetch on the server, test header case sensitivity and redirect behavior vs browser fetch.

- [P-155] **[P2]** Firefox 152 Experimental JPEG XL Support — Next-Gen Image Format Creates Browser-Specific Rendering (so.html5.qq.com/page/real/search_news?docid=70000021_9116a30cb5d40652 Jun 16 2026 — "Mozilla 更新推出火狐 Firefox 浏览器 152 稳定版,聚焦改版设置页面､增强媒体控制､实验性支持 JPEG XL 等"). **Why this is the NEXT-GEN-IMAGE-FORMAT dimension distinct from P-154 (Temporal API — date/time) and P-149 (iOS WebKit performance)**: P-154 is date/time; P-149 is mobile engine speed; P-155 captures the NEW-IMAGE-CODEC-SUPPORT class where Firefox 152 adds experimental JPEG XL support — JPEG XL offers ~30% better compression than JPEG/WebP, but support across browsers is inconsistent (Chrome removed JXL support in 2023, Safari has partial/flagged support, Firefox 152 enables it experimentally). **Why this matters for GameZipper**: (a) if any game assets (backgrounds, textures, sprite sheets, OG images) are ever converted to JPEG XL for size savings, they will render in Firefox 152+ but show broken-image icons in Chrome and most Safari versions — creating a browser-specific visual breakage; (b) the `<picture>` element with `<source srcset="image.jxl" type="image/jxl">` fallback to `<img src="image.jpg">` must be used for progressive enhancement, but if a game hardcodes `.jxl` URLs in canvas `drawImage()` or CSS `background-image: url(*.jxl)`, there's no fallback; (c) `Image.decode()` and `createImageBitmap()` may reject JPEG XL silently in non-supporting browsers, causing game initialization to hang if the error isn't caught; (d) feature detection for image format support is non-trivial — the standard approach is `document.createElement('canvas').toDataURL('image/jxl')` but this checks ENCODING not DECODING; the correct check is loading a tiny JXL and seeing if `img.decode()` resolves; (e) experimental support means Firefox may disable it in a future point release, breaking games that shipped JXL assets assuming stable support. **QA methodology**: (1) audit all image assets: `grep -rn '\.jxl' --include="*.html" --include="*.js" --include="*.css"` — should be 0 for production safety; (2) if JXL is desired, use `<picture>` with type fallback AND verify the fallback JPG/WEBP loads correctly in Chrome/Safari; (3) for canvas-based games, test `new Image(); img.src = 'test.jxl'; img.decode().then(...).catch(...)` — the catch path must provide a fallback; (4) check `Content-Type: image/jxl` is served by the CDN for `.jxl` files; (5) if using OG images for social sharing, verify that social platforms (Facebook, Twitter/X, Discord) support JXL — most do NOT, so OG images must remain JPG/PNG.
