B-201 captures the HARDWARE-REFRESH-MISMATCH class where 2026 gaming phones (iQOO, RedMagic, ROG Phone) ship with 144Hz displays and 3000Hz touch sampling rates — but most browser games are hardcoded for 60fps (16.67ms frame budget) and process touch events at browser-throttled rates. **Why this matters for GameZipper**: (a) a 144Hz display refreshes every 6.94ms — a 60fps game loop only updates every 16.67ms, meaning the display shows the SAME frame 2-3 times, creating visible judder/micro-stutter that doesn't exist on 60Hz displays; (b) 3000Hz touch sampling means the phone detects touch events every 0.33ms — but browsers throttle `touchmove` events to frame rate, so fast swipes may lose intermediate points, creating jagged lines or missed gestures; (c) `requestAnimationFrame` on a 144Hz display fires at 144fps — game loops that hardcode `dt = 16.67` (instead of computing actual delta time) will run 2.4x TOO FAST on these phones; (d) games using `e.timeStamp` or `performance.now()` for input timing may see sub-millisecond precision that breaks integer-based game logic; (e) the "super-frame concurrent" feature means the phone's display chip may interpolate frames — a game that intentionally shows a blank frame (screen flash) may have the blank frame interpolated away. **QA methodology**: (1) test 5+ games on a 144Hz Android phone, verify game speed is correct (not 2.4x); (2) check all `requestAnimationFrame` callbacks compute `dt` from timestamps, not hardcode 16.67; (3) test swipe/draw games at high touch rate, verify no lost points; (4) verify screen-flash effects are visible (not interpolated).
