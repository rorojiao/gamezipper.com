W-346 captures the BFCACHE-NOW-INCLUDES-WS-PAGES class where Chrome 149 changed bfcache eligibility: previously, pages with live WebSocket connections were EXCLUDED from bfcache (the open socket made the page un-cacheable). Chrome 149 now auto-closes the WS before saving to bfcache and lets the page qualify for instant restore — when the user navigates back, the page is restored INSTANTLY (no reload, no re-init), but the WS is GONE (closed). The page must detect the loss and reconnect. **Why this matters for GameZipper**: (a) GameZipper's `gz-analytics.js` (or any in-house analytics lib) may open a long-lived WebSocket / EventSource for telemetry — Chrome 149 will auto-close it on back-nav, and the analytics will silently stop recording until reconnect logic kicks in (which most naive implementations never do); (b) any game that opens a WebSocket for cloud-save sync / multiplayer state / leaderboard updates will silently lose the connection after back-nav, causing "my score didn't save" or "leaderboard froze" bugs that look like network issues but are actually bfcache; (c) the bfcache RESTORE (pageshow with persisted:true) is INSTANT — the game resumes from saved state but with no live data feed, so UI elements showing "live X players online" will show stale data forever until the user navigates again; (d) GameZipper should listen for `pageshow` event with `event.persisted === true` and re-establish WS connections on back-nav restore; (e) the existing GameZipper pitfall around `defer` script timing (f2bd0b32 fix) is COMPOUNDED by bfcache — a bfcached page has its defer scripts already executed, so any re-init logic must run on pageshow, not on DOMContentLoaded. **QA methodology**: (1) open a GameZipper game, wait for WS-connected HUD element to appear, navigate to another page, hit back, observe whether the HUD shows live data or stuck stale value; (2) instrument: `window.addEventListener('pageshow', e => console.log('persisted:' + e.persisted))` — if e.persisted is true, the page came from bfcache; (3) verify GameZipper analytics endpoint re-establishes on pageshow/persisted; (4) Chrome 149 stable rolled out Jun 2 2026 — by mid-July 2026 the bulk of GameZipper users (Chrome dominant browser) are on Chrome 149+, so this is an immediate real-world regression risk.
