### Performance (P)

- [P-149] **[P1]** iOS WebKit-Only Engine Restriction — 28.6% Performance Penalty vs Chromium Browsers (Microsoft Speedometer 3.1 Benchmark) (so.html5.qq.com/page/real/search_news?docid=70000021_8006a3327f492452 Jun 18 2026 — "微软工程师公布了一组基准测试数据:在苹果自家iOS系统的测速工具Speedometer 3.1性能测试中,一款基于Chromium且采用微软自有渲染引擎的浏览器在iOS系统上的得分比Safari高出28.6%"). **Why this is the iOS-WEBKIT-MONOPOLY-PERFORMANCE-PENALTY dimension distinct from B-198 (Firefox UI density — content area sizing) and P-147 (WebGPU+WASM pipeline — rendering performance)**: B-198 is about Firefox content area sizing; P-147 is about GPU+WASM rendering performance on DESKTOP; P-149 captures the iOS-PLATFORM-RESTRICTION class where Apple's App Store policy mandates ALL iOS browsers (Chrome, Firefox, Edge, Opera — not just Safari) MUST use the WebKit engine under the hood — there is NO "Chrome on iOS uses Blink" — Chrome on iOS IS Safari with a Chrome skin. **Why this matters for GameZipper**: (a) ~30% of GameZipper mobile traffic is iOS — ALL of those users are capped at WebKit's JS execution + rendering performance regardless of their browser choice — physics-heavy games (block stacking, water sort solver, ragdoll) run 28.6% slower on iOS than on equivalent Android hardware; (b) canvas rendering, WebGL, and requestAnimationFrame timing are ALL gated by WebKit's engine — GameZipper cannot optimize for "Chrome on iOS" because it doesn't exist as a separate engine; (c) the 28.6% gap means games tuned to run at 60fps on desktop Chromium may drop to ~43fps on iOS WebKit — frame-rate-dependent game logic (collision, physics stepping) behaves differently, potentially causing gameplay bugs (tunneling through walls, missed collisions); (d) WebKit's garbage collector and JIT compiler have different performance characteristics than V8 (Chrome desktop) — games with high object allocation (particle systems, many short-lived game entities) may stutter more on iOS; (e) the EU Digital Markets Act may eventually force Apple to allow non-WebKit browsers on iOS — GameZipper should prepare for a FUTURE where iOS Chrome/Firefox use Blink/Gecko and perform differently from iOS Safari. **QA methodology**: (1) benchmark GameZipper physics-heavy games on real iOS devices (iPhone 12/13/14/15) measuring FPS via `requestAnimationFrame` timestamp logging — compare against desktop Chrome at 60fps; (2) verify game logic uses delta-time-based physics stepping (`dt = now - lastTime`) NOT frame-count-based — delta-time ensures consistent physics regardless of FPS; (3) test on iOS Safari 17+ AND iOS Chrome (same engine) to confirm identical behavior; (4) for WebGL games, check if iOS WebKit supports the same WebGL extensions as desktop — iOS may lack some extensions causing fallback rendering; (5) monitor iOS crash logs (Safari WebKit crashes) via Safari Web Inspector for games that push performance limits. **Severity**: P1 — affects ~30% of mobile traffic, causes measurable gameplay degradation on iOS.

- [P-150] **[P2]** Chrome DevTools Heap Snapshot + Allocation Timeline — Detached DOM + Layout Thrashing Memory Leak Detection (cloud.tencent.com/developer/article/2681608 Jul 11 2026 — "前端硬核调试:基于Chrome DevTools定位复杂内存泄漏与渲染瓶颈的实战指南...深入Chrome开发者工具底层,解析堆快照(Heap Snapshot)与分配时间轴的调试机制,通过真实前端代码示例,拆解如何定位'分离DOM'与'强制同步布局(Layout Thrashing)'两大核心痛点"). **Why this is the DEVTOOLS-DEEP-MEMORY-PROFILING dimension distinct from P-057 (WASM memory OOM — failure mode) and the BGM/setInterval leak pattern (game-footer skill pitfall)**: P-057 is a WASM-specific OOM; the BGM leak is an untracked interval ID; P-150 captures the GENERAL-MEMORY-LEAK-PATTERN class with two sub-patterns: (1) **Detached DOM** — game elements removed from the DOM tree via `removeChild()` or `innerHTML = ''` but STILL REFERENCED by JavaScript closures, event listeners, or data structures — the browser cannot garbage-collect them, causing invisible memory growth proportional to game session length; (2) **Layout Thrashing** — interleaving DOM read operations (`element.offsetWidth`, `getBoundingClientRect()`) with DOM write operations (`element.style.width = ...`) forces the browser to recalculate layout SYNCHRONOUSLY for each read-after-write cycle — in a game loop running 60fps, this causes severe frame stutter. **Why this matters for GameZipper**: (a) games that create/destroy DOM elements per frame (particle effects using divs, dynamic UI updates, score popup elements) accumulate detached DOM nodes — after 10 minutes of play, memory may grow by 50-100MB, eventually causing tab crashes on mobile; (b) games that read canvas/game-element dimensions inside the render loop and then modify styles trigger Layout Thrashing — the main thread blocks on layout calculation 60 times/second; (c) the detached DOM pattern is INVISIBLE to casual testing — the game works fine for 2-3 minutes but degrades over 10+ minutes — QA must use Chrome DevTools Memory tab to detect it; (d) the `game-footer.js` + `monetag-manager.js` scripts create DOM elements that may be detached if not properly cleaned up on page unload. **QA methodology**: (1) open Chrome DevTools → Memory tab → take Heap Snapshot BEFORE playing a game; (2) play the game for 5 minutes (interact, start/restart levels, trigger particle effects); (3) take another Heap Snapshot → compare → look for "Detached" DOM trees in the comparison view — any detached element with retained size >1KB is a leak; (4) use Allocation Timeline to record 60 seconds of gameplay — look for continuous allocation spikes that never get garbage-collected (sawtooth pattern that trends upward = leak); (5) for Layout Thrashing: open Chrome DevTools → Performance tab → Record 5 seconds of gameplay → look for "Recalculate Style" and "Layout" events taking >5ms each — if they appear EVERY frame, the game has Layout Thrashing; (6) fix detached DOM by nullifying references (`element = null`) after removal and using WeakRef where appropriate; (7) fix Layout Thrashing by batching DOM reads BEFORE all writes in each frame (`var widths = elements.map(e => e.offsetWidth); widths.forEach((w, i) => elements[i].style.width = w + 'px')`). **Severity**: P2 — causes gradual degradation, affects long play sessions.
